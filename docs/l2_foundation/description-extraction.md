## 1. 機能概要

### 1.1 この機能は何か

**動画説明文からの自動タグ抽出機能**である。ニコニコ各シリーズの説明文（description）は、あらすじだけでなく
**キャスト・スタッフ・制作会社・著作権表記**を 1 本の文字列に詰め込んで配信される。この機能はその文字列を
機械的に解析して **関係者の名前（声優・監督・脚本・音楽・原作者・制作会社など）を抽出**し、
**演者/制作を区別しない 1 列の発見タグ**として作品詳細画面に表示する。各タグは正規化キーを持ち、
クリックすると**同じ人物/会社が関わる他作品の一覧へ横断フィルタ**できる。

公式に構造化されたクレジット API は存在せず、説明文という非定型テキストが唯一の情報源であるため、
**テキスト解析による自動抽出**という形をとる。

### 1.2 ユーザーから見た振る舞い

機能の外形仕様（実装は `web/src/features/detail/detail.ts` `buildCredits` / `web/src/features/list/filter.ts`）:

| 振る舞い                        | 仕様                                                                                                                                                                                                                                                                                                                |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **詳細画面に「演者/制作」1 行** | 声優・スタッフ人名・制作会社・原作者を **1 カテゴリに統合**したタグ列を表示。**名前のみ**（役名・役割ラベルは出さない）。                                                                                                                                                                                           |
| **全タグ均一にクリック可能**    | 全 credit タグは**既存タグ `#…` と同じ作法で必ずクリック可能**。クリックで `?credit=<canonical key>` へ遷移し、一覧をその key を含む作品に絞る（完全一致）。**出現回数で見た目・クリック可否・並びを変えない**（予測どおりに動く）。1 作品しか該当しない人物でも普通にクリックでき、結果がその 1 件出るだけ＝一貫。 |
| **重複除去**                    | 同一 canonical key は 1 つにまとめる（表記ゆれも吸収）。                                                                                                                                                                                                                                                            |
| **長い名前は省略**              | 20 文字超のチップは末尾省略＋全文ツールチップ。                                                                                                                                                                                                                                                                     |
| **自動抽出の注記 (i)**          | 見出しに (i)。自動抽出ゆえ誤り・抜けがある旨を明示（`CREDIT_NOTE`）。                                                                                                                                                                                                                                               |

### 1.3 設計の要点（旧 cast/staff 分類からの変更）

旧 PH-0014 は cast/staff/studios/copyright を **2 カテゴリに分類**していたが、発見用途
（同じ人で他作品を探す）では cast/staff の区別は無意味で、分類の脆さ・表記ゆれ・copyright 埋没で
取りこぼしが多かった。本設計は:

- **1 カテゴリ統合**: バケツ（cast/staff）を捨て、全関係者を 1 列に。混在段落バグが消える。
- **canonical key 正規化**: `諏訪部 順一` と `諏訪部順一` を 1 実体に集約（§6）。
- **全タグ均一クリック可・noise は抽出で落とす**: 出現回数（recurrence）で表示やクリック可否を
  差別化しない（操作の一貫性・予測可能性のため）。代わりに **noise（曲名・◯◯製作委員会・©/年号・
  役名・汎用語）を抽出ルール側で除去**し、画面に出るタグが全て「クリックする価値のある実在の
  人物/会社名」になっている状態を作る（§7）。
- **soft provenance**: `source`(castLike/staffLike/studio/copyright/themeSong)・`role` は内部保持のみ
  （抽出可否の gate には使わない・将来の序列/facet 用）。

> **recurrence の位置づけ**: 「同一 key が何作品に出るか」の集計（`countRecurrence`）はユーティリティ
> として残すが、**UI の表示・クリック可否・並びには一切使わない**。`?credit=` の索引は works.json
> （key→作品集合）そのものが担う。

## 2. 評価軸（なぜこの設計か）

ユーザーの動機は「**同じ制作スタジオ・監督・脚本・声優などで他のアニメ等を探す**」。よって抽出物は
「**人物/会社の横断発見タグ**」として役立つかで評価する。役名・「誰が何役か」はどうでもよく、
**名前がクリックして他作品に飛べるか**が価値。この軸から:

1. **分類より正規化** — cast/staff のラベルは join に無関係。重要なのは表記ゆれを潰して同一実体に集約。
2. **連結値は割る・copyright は救出する** — 「忠実だが一致しない blob」より「飛べるタグ」。
3. **出すタグは実在名だけ** — クリック可否を出現回数でゲートせず（一貫性）、noise は抽出段で落とす。
   「1 作品しか該当しない」ことはブロック理由にならない（クリックすれば 1 件出る＝正しい挙動）。

## 3. 入力 — 何をパースするか

抽出処理（`extractCredits`）に渡すのは、**各シリーズの「1話目（最古話）」の説明文 1 本だけ**である
（`store.mjs` `_buildSeriesJson` / `credit-index.mjs` `buildCreditIndex`）。

- シリーズ全話ではなく `chronoSort` 昇順の先頭＝最古話 1 件のみ。関係者はシリーズ内でほぼ一定なので 1話目で十分で、
  あらすじ（`descriptionFirst`）と同じ話を源にすることで源不整合を避け、全話パースのコストも省く。
- その 1 本は**源優先マージ（`chooseDescription`）の勝者**（①構造 `<br>` → ②源 `snapshot>nvapi>rss` → ③長さ）。
- **入力形式の非依存**: `extractCredits` は生 HTML（`<br>` 付き・fetch 直後）でも、stored の stripHtml 済み
  （`<br>`→`\n\n` 化済み・`data/series/*.json`）でも読める。構造判定は **`<br>` または `\n\n` の有無**
  （`isStructuredCredits`）。旧版は `<br>` の存在に依存していたため毎時 partial で credits を再計算できなかったが、
  本版は `\n\n` でも読めるので **毎時でも 1話目から再抽出できる**（carry-forward 不要）。

## 4. 元データの実態（事実）

### 4.1 構造化版とフラット版

| 書式           | 由来             | 中身                                                                      | 扱い                                 |
| -------------- | ---------------- | ------------------------------------------------------------------------- | ------------------------------------ |
| **構造化版**   | snapshot / nvapi | あらすじ・キャスト・スタッフ・© を `<br><br>`（stored は `\n\n`）で区切る | 段落に割って抽出                     |
| **フラット版** | チャンネル RSS   | 全要素を区切り無しで連結（`<br>`/`\n\n` 無し）                            | **分解しない**（境界不明＝誤検知源） |

構造化版がほぼ全話（`\n` を持つ description が 99.5%）。フラットは新着回転窓のみ。

### 4.2 クレジット段落の形

```
（あらすじのプロ―ズ）

役名:声優A／役名:声優B …            ← cast ブロック
原作:○○／監督:△△／制作:□□ …       ← staff ブロック
©○○製作委員会                       ← copyright（制作実体が紛れることがある）
so… ←前話 ／ 次話→ so…               ← 話リンク
```

クレジット行は **全角 `／` でエントリを並べ、各エントリは `役:値`（コロン区切り）**。半角 `:`・全角 `：` 両対応。
舞台/ライブの出演は **全角スペース `　`(U+3000) で名前を区切る**（半角スペースは姓名内＝`木村 了`。データ規約で確認）。

### 4.3 種別ごとの違い（種別専用パーサは持たない）

- **アニメ・映画**: `役名:声優` がきれい。よく取れる。
- **舞台（2.5次元）**: `出演:キャラ名 役：俳優名` 形式 → **`役：`（または末尾コロン）の後ろの俳優名だけ救出**。
- **ライブ**: 曲順 setlist（`01:曲名`）は role が数値で落とし、アーティスト/作曲人名は拾う。
- **混在段落**: 映画等で `出演:声優…／原作:…／制作:…` が 1 段落に同居（おしりたんてい型）。
  **per-segment 分類**で声優も制作も取りこぼさず拾う。

## 5. 抽出パイプライン

`extractCredits(rawHtml)` は次の段で動く（すべて `scripts/etl/credits.mjs`）。返すのは
`{ structured, tags: [{display,key,source,role}], synopsis, copyrightRaw }`。

### 5.1 構造判定 → 段落分割

`stripHtml` で正規化（`<br>`/`</p>` 等→`\n`、実体参照デコード、連続改行畳み込み）→ `\n{2,}` で段落配列に割る。
`isStructuredCredits` が false（フラット）なら**一切分解せず** synopsis に温存。

### 5.2 段落分類（per-segment）

`／` で割った各セグメントを**個別に**分類する（ブロック丸ごとの cast/staff 振り分けはしない＝混在段落対策）。

1. **話リンク / info**（`←前話`/`次話→`/`動画投稿`等）→ 捨てる。
2. **copyright**（`©`/`製作委員会` 等を含む段落）→ 生文字列を `copyrightRaw` に温存しつつ、
   **`役:値` 構造があれば再パースして制作実体（原作者/監督/制作会社）を救出**。
3. **プロ―ズ段落**（`hasProsePeriod`＝文末の `。`）→ あらすじ/各話概要なので**抽出しない**
   （`第1話：…／第2話：…` の誤検知防止）。混在クレジットは文末を持たないので通る。
4. **その他のコロン段落** → `parseBlock` で `／` 分割 → 各セグメントを `役:値`（最初のコロン）に割り、
   `entryToTags` で role を見て振り分ける。割れないセグメントは**そのセグメントだけスキップ**（lossless）。

### 5.3 セグメント → タグ（`entryToTags`）

各 `役:値` セグメントの **value 側だけ**をタグ化する（role＝役名/役割は内部保持のみ）。role を見て:

- **studio role**（`制作`/`製作`/`アニメーション制作`等・`STUDIO_ROLE_RE`）→ value は制作会社。`source=studio`。
- **staff role**（`STAFF_KEYS` の語: 原作/監督/脚本/音楽/作画監督/演出…）→ value は人名。`source=staffLike`。
- **song role**（`主題歌`/`ED`/`劇中歌`等・`SONG_ROLE_RE`）→ **曲名（`「」『』`）を捨て、`作詞/作曲/編曲/歌：人名`
  を二次パースして救出**（`extractSongNames`）。`source=themeSong`。劇伴 `音楽=○○` は song 扱いせず人名で取る。
- それ以外（混在ブロックの声優等）→ value は人名。`source=castLike`。

value の整形順（重要）:

1. **引用符タイトル除去**（`「」『』`＝作品/曲名）。英字タイトル `『FAIRY TAIL 100 YEARS QUEST』` が
   後段のスペース分割で語に砕けるのを防ぐ。括弧 `（）`＝所属は残す。
2. **エントリ分割**: 全角スペース `　`(U+3000)・読点で分割（舞台の `A　B　C`）。
   日本語名で**括弧外の半角スペースが 3 名以上**なら半角スペースでも分割（`関根優那 高橋りな 寒竹優衣…`）。
   `木村 了`（1 スペース）やラテン社名（`Planet Kids Entertainment`）は割らない。
3. **末尾コロン取り**: `キャラ（/別名）役：俳優` は最後のコロン以降＝実在名を採る（役名/ユニット名は捨てる）。
4. **連結分割**（`splitConnected`）: 中黒 `・`・スラッシュ `/／`（括弧外のみ）。Western 名 `リリー・フランキー`
   （全カタカナ）・頭文字名 `M・A・O`（全 1 字）・denylist（`Wake Up, Girls!`）は割らない。
5. **所属括弧分離**（`splitAffiliation`）: `米山和仁（劇団ホチキス）`→人物＋会社。出版社注記
   （`集英社…連載`）は捨ててクリーン名（`岸本斉史`）。
6. **正規化 → key**（§6）。noise は §7 のガードで弾く。

### 5.4 救出される代表ケース

- **copyright 強奪の解消**: 響け！ユーフォニアム `…／監督:石原立也／…／製作:『響け！』製作委員会` →
  再パースで **石原立也・山田尚子・京都アニメーション**を救出（委員会名はノイズで落ちる）。
- **混在段落**: おしりたんてい映画の声優（三瓶由布子…）＋監督・制作会社・主題歌作曲を全部拾う。
- **連結値**: `原作:奈須きのこ・TYPE-MOON`→2 タグ。`須藤友徳・田畑壽之・碇谷敦`→3 タグ。
- **舞台**: `出演:小野田坂道 役：糠信泰州、…`→俳優名。`出演:木村 了　入江甚儀　三津谷 亮…`→各俳優名。

## 6. 正規化（canonical key）

各 value を**表示名（display）と canonical key**の 2 層にする。**フィルタは key 完全一致**、表示は display。
これが「同一人物を違う表記でも 1 クリックに集約」の核心。

| 処理               | 関数                 | 例                                                                                                                                     |
| ------------------ | -------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| canonical key 生成 | `normalizePersonKey` | NFKC（`ＭＡＰＰＡ`→`mappa`）＋内部空白除去（`諏訪部 順一`→`諏訪部順一`）＋法人格除去（`株式会社トレノバ`→`トレノバ`）＋ latin 小文字化 |
| 表示名整形         | `cleanDisplay`       | 先頭 `©` 除去（`©Frontwing`→`Frontwing`）・括弧/引用符以降除去・末尾 `他`/`※`/`著`/`…`/年 除去                                         |

**順序の鉄則**: 引用符タイトル除去 → エントリ分割（全角/括弧外半角スペース）→ 末尾コロン取り →
連結分割（括弧外）→ 所属分離。これにより `今野康之（スワラ・プロ）` の括弧内 `・` で割らず、
`真島ヒロ・上田敦夫『英字タイトル』` の英字が砕けない。

## 7. noise 除去（出すタグを実在名だけにする）

全タグを均一に表示・クリック可にするため、**名前/会社でないものは抽出段で確実に落とす**。

| #   | 落とすもの                       | どう落とすか                                                                                                    |
| --- | -------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| ①   | フラット RSS の連結文字列        | `<br>`/`\n\n` 構造が無ければ**一切分解せず** synopsis（`isStructuredCredits`）                                  |
| ②   | 各話概要・あらすじ（`第1話：…`） | 段落が**プロ―ズ文末 `。` を持てば抽出しない**（`hasProsePeriod`）＋ role が `第N話`/`#3`（`EPISODE_MARKER_RE`） |
| ③   | setlist 曲順（`01:曲名`）        | role が数値/記号のみなら不採用（`NUMERIC_ROLE_RE`）                                                             |
| ④   | 製作委員会名・年号 copyright     | `isCommittee`（製作委員会/パートナーズ）・`©…20xx` は生値で弾く。先頭 © は除去（`©Frontwing`→`Frontwing`）      |
| ⑤   | 放送局・役割語・汎用語           | 放送局（`BROADCASTER_RE`）・`他`/`ナレーション`/`作曲`/`歌`/`and more` 等（`VALUE_STOPWORDS`）                  |
| ⑥   | 曲名・作品名・掲載誌注記         | 引用符 `「」『』` 内は除去（§5.3-1）。出版社注記（`集英社…連載`）は `splitAffiliation` で捨てる                 |
| ⑦   | 注記文・プロ―ズ片                | `です`/`ます`/`表記`/`について` を含む value、文末 `。`、長すぎ（>40）は不採用（`isPlausibleName`）             |
| ⑧   | 頭文字片・1 文字キー             | 正規化後 1 文字の key は不採用（`M・A・O` は割らないので 1 名で残る）                                           |

**監査（2026-06-29 実測）**: 全カタログ 6,601 シリーズ・抽出タグ約 14.3 万件・distinct key 約 3.1 万。
「括弧/句読点/委員会/年号/役割語/1 文字」で機械監査した結果、真の noise は数十 key（≈0.05%）まで圧縮。
残りの「疑い」はほぼ実在の長い社名（`Planet Kids Entertainment` 等）・かな名・スタジオ（`project No.9` 等）の
誤検出（＝実害なし）。再現は `scratch/noise-all.mjs`、代表ダンプは `scratch/credit-dump.md`。

## 8. データモデルと配線

```
1話目 description → extractCredits（§5）→ tags[{display,key,source,role}]
        ↓ buildCreditIndex（seriesId→tags の per-series 索引）
        ├→ series JSON  credits: CreditTag[]   （seriesCredits・{name,key,source,role}・抽出順・key dedup）
        └→ works.json   credits: string[]      （worksCreditKeys・全 canonical key・dedup）
                                   ↓ web
   詳細画面 buildCredits（全タグ <a> で均一クリック可）→ `?credit=<key>`
   一覧 filter（state.credit を work.credits[key] に完全一致）
```

- **series JSON** `credits`: `{name, key, source, role}` の配列（抽出順・recurrence で並べ替えない）。
- **works.json** `credits`: 全 canonical key の配列（`?credit=` 照合用。singleton キーも含む＝クリックで自作品が出る）。
- **web 型**: `CreditTag`（`web/src/data/types.ts`・`{name, key, source?, role?}`）。詳細は
  `SeriesDetail.credits: CreditTag[]`、一覧フィルタは `Work.credits: string[]`（key）。URL は `?credit=<key>`。
- **後方互換**: 旧 JSON（`credits: string[]`）は web 側 `normalizeCreditTags` が `{name,key}` に正規化。

## 9. カバレッジ実測

**測定法**: ローカル全 6,601 シリーズ（epCount>0）の 1話目 stored description に `extractCredits` を適用し、
**「クリック可能な credit タグが ≥1 個取れるシリーズ率」**を測った（`scratch/measure-coverage.mjs`・
`scratch/genre-coverage.mjs`、2026-06-29 実測。全タグがクリック可なので「抽出タグ ≥1」と同義）。

| 指標                                | 値                  |
| ----------------------------------- | ------------------- |
| 全シリーズ（epCount>0）             | 6,601               |
| **≥1 クリック可能タグ（要件≥70%）** | **6,512（98.65%）** |
| distinct canonical keys             | 約 33,000           |
| クリック可能タグ数 中央値           | 約 22               |

**100% にならない 89 件（1.35%）の内訳**（取りこぼし＝バグではないことを実測で確認・`scratch/audit-coverage.mjs`）:

| 区分                                      | 件数  | 正体                                                               |
| ----------------------------------------- | ----- | ------------------------------------------------------------------ |
| フラット（`<br>`/`\n\n` 無し・RSS 等）    | 30    | 構造が無く分解不能（次 daily で snapshot 構造版に置換）            |
| 構造あり・人物/会社名が元データに無い     | 59    | あらすじのみ・setlist のみ等＝**取りこぼしでなく元から名前が無い** |
| 構造あり・クレジット様だが0＝**抽出バグ** | **0** | —                                                                  |

ライブ/舞台もプロ―ズ判定等のバグ修正後はクリーンに取れるため**除外しない**（全種別を credit 対象に含める）。

### recurrence は表示に不使用（「1出現で即タグ」）

クリック可否・表示・並びは出現回数に **一切依存しない**。各シリーズ1話目から抽出した時点で
**1出現でも即タグ化＆クリック可能**。works.json の credits は全 canonical key（singleton 含む）なので
「2件になったら昇格」のような daily 再計算依存の挙動は**存在しない**。落とすのは §7 の noise（名前でないもの）だけで、
**実在の人物/会社名は1作品しか出ないものも含め1つも落とさない**（`scratch/credit-dump.md`・`audit-precision.mjs` で確認）。

## 11. 既知の制約

- **出演がグループ名のみ**（ライブ）: `出演:JAM Project` 等はそのグループ名 1 タグになる（メンバー個別が書かれていれば拾う）。
- **区切り無しの連名**: `阿部智佳子（タバック）今井修治` のように区切りの無い 2 名は分割不能で 1 タグ。
- **役割ラベル（role）/ source**: 内部保持のみ・UI 非表示（曖昧な役名の露出回避・将来 facet 用 soft metadata）。
- **横断 facet（声優別/制作別索引）**: 未実装。`countRecurrence`（key→件数）や works.json（key→作品集合）が将来の素。
- **事実の正しさ**: §10 の射程どおり、名前単位の事実誤り・取り違えは検証対象外。
