## 1. 機能概要

### 1.1 この機能は何か

**動画説明文からの自動タグ抽出機能**である。ニコニコの各シリーズの動画説明文（description）は、あらすじだけでなく**キャスト・スタッフ・制作会社・著作権表記**を 1 本の文字列に詰め込んで配信される。この機能は、その文字列を機械的に解析して

- 声優名（cast の値）
- スタッフ人名（監督・原作・脚本・音楽・作詞・作曲・演出 等の値）
- 制作会社名（アニメーション制作 等）
- copyright 行から掘り出した制作実体（原作者・出版社・スタジオ等）

を、**1 つの dedup した「名前タグ」列に統合して抽出**し、作品詳細画面に「演者/制作」として表示する。さらに各名前タグをクリックすると、その人物/会社が関わる作品の一覧へ絞り込める（`?credit=<名前>`）。

**なぜ 1 カテゴリに統合するか。** この機能の用途は「**同じ声優・監督・脚本・音楽・制作スタジオ・原作者などで他作品を探す**」ことに尽きる。発見用途では「これは声優か、スタッフか、会社か」の区別は不要で、**他作品に繋がる名前であるか**だけが価値を決める。旧設計は cast / staff の 2 カテゴリに分類していたが、

- 映画等では声優が staff 塊に混入する分類バグが起きた（例: 『おしりたんてい』で `おしりたんてい:三瓶由布子／スイセン:仲里依紗` は声優、以降 `原作:トロル／監督:セトウケンジ／…/制作:東映アニメーション` がスタッフ、という塊の取り違え）。
- 区別が要らない用途で分類を維持すること自体が脆さとカバレッジ低下の原因だった。

そこで分類をやめ、全関係者を 1 列に統合する。これで分類バグが構造的に消え、copyright マイニングと合わせて recall も上がる。

公式に構造化されたクレジット API は存在せず、説明文という非定型テキストが唯一の情報源であるため、**テキスト解析による自動抽出**という形をとる。自動抽出ゆえに必然的に不正確を含むので、本機能は「**構造的に妥当な名前だけを出す（プロ―ズ・台詞・曲順番号などの誤抽出ゼロ）**」を品質の主眼とし、確実に解釈できないものは抽出しない（→ §1.5・§5・§9）。

### 1.2 ユーザーから見た振る舞い

機能の外形仕様（実装は `web/src/features/detail/detail.ts` `buildCredits` / `web/src/features/list/filter.ts`）:

| 振る舞い                          | 仕様                                                                                                                                                                    |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **詳細画面に「演者/制作」が出る** | 作品詳細に「演者/制作」**1 行**を表示。声優・スタッフ人名・制作会社・原作者等を区別せず **名前のみ**のタグ列で並べる（役名・役割ラベルは表示しない）。                  |
| **空なら出さない**                | 名前が 1 つも取れなければクレジット欄ごと非表示。                                                                                                                       |
| **クリックで一覧フィルタ**        | 各タグはクリックでその名前の作品一覧へ遷移（`?credit=<名前>`）。一覧は `work.credits` にその名前を含む作品だけに絞り込む（完全一致）。既存のタグチップと同じ作法。      |
| **重複除去**                      | 同名は 1 つにまとめる（声優と制作会社が同名でも 1 つ）。                                                                                                                |
| **長い名前は省略**                | 20 文字超のチップは末尾省略＋全文ツールチップ。                                                                                                                         |
| **自動抽出の注記 (i)**            | 「演者/制作」見出しに (i) ボタンを置き、ツールチップで「これらは説明文からの自動抽出のため、誤り・抜け・不正確が含まれる場合があります。」と明示する（`CREDIT_NOTE`）。 |

この「不正確がありうる」注記は**仕様の一部**である。誤抽出を抑える代わりに recall は犠牲にしている（取りこぼしがある）ため、欠落・不正確の可能性をユーザーに開示する。

### 1.3 用語定義

| 用語                       | 定義                                                                                                              |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| **説明文 / description**   | 各話に紐づく動画説明文（1 本の文字列）。あらすじ・クレジット・©・話リンクを含む。                                 |
| **構造化版**               | 要素を `<br>` で区切った説明文（snapshot / nvapi 由来）。分解できる。                                             |
| **フラット版**             | 全要素を区切り無しで連結した説明文（RSS 由来）。分解しない。                                                      |
| **credits（演者/制作）**   | 関係者の名前を 1 列に統合・重複除去した名前タグ配列。本機能の出力。`?credit=` の照合対象。                        |
| **copyright マイニング**   | © 行から制作実体（原作者・出版社・スタジオ等）の名前を掘り出すこと。©記号・年・製作委員会名は捨てる。             |
| **recurrence（再登場性）** | その名前が他作品でも現れるか。本質的に 1 作品しかヒットしない名前（曲名・役名・年・製作委員会名）は発見に無価値。 |
| **誤抽出ゼロ**             | 出した名前が**構造的不変条件**を破らない（プロ―ズ・台詞・曲順番号の混入なし）。正解突合とは別＝§9。               |

### 1.4 入力 / 出力

- **入力**: 各シリーズの「1話目（最古話）」の説明文 1 本（→ §2）。
- **抽出処理**: ETL（`scripts/etl/description.mjs` `extractCredits`）が名前を統合抽出する（→ §4）。内部で `parseDescription` が段落分類（cast/staff/copyright/synopsis）まで行い、`extractCredits` がそこから名前だけを掃除・統合する。
- **出力（データ）**: SeriesEntry / WorkEntry の `credits`（名前配列・series 単位・1話目由来・重複除去）。型定義の正本は [`data-inventory.md`](data-inventory.md)。
- **出力（画面）**: 詳細画面の「演者/制作」タグ列と、`?credit=` による一覧フィルタ（→ §1.2）。

### 1.5 品質要件とスコープ外

**品質要件**:

- 出した名前タグは **構造的に妥当なものだけ**（プロ―ズ・台詞・曲順番号・曲名・製作委員会名・年などの誤抽出はゼロ。全数の不変条件監査で実証）。ただし「全数を正解データと事実突合した」わけではない点は §9 で正直に明示する。
- 確実に解釈できないものは抽出せず、原文（synopsis）に温存する（lossless）。
- recall（網羅率）は犠牲にしてよい。取れないものは取らない。
- カバレッジ（各シリーズ 1話目から ≥1 名取れる率）は **7 割以上**を必達とする（実測 **99.0%** ＝ §6）。

**この機能が扱わないこと**:

- **どの説明文を入力に選ぶか**（snapshot / nvapi / RSS のどれを採るか＝源優先マージ）は別トピック。本書は「選ばれた 1 本をどう名前に落とすか」だけを扱い、源の選定は §2 で 1 段落だけ触れて [`dataflow.md`](dataflow.md) に委ねる。
- **credits のグローバル facet 化**（声優別・制作別の横断インデックス）は未実装（名前クリックの単発フィルタのみ）。
- **役名・役割の区別/表示**（声優か監督か会社か）は持たない。発見用途で不要なため統合した（§1.1）。

## 2. 入力 — 何をパースするか

抽出処理（`extractCredits`）に渡すのは、**各シリーズの「1話目（最古話）」の説明文 1 本だけ**である。

- シリーズの全話をパースするのではなく、`chronoSort` 昇順で先頭＝最古話 1 件のみ。出演/制作はシリーズ内でほぼ一定なので 1話目で十分であり、あらすじ（`descriptionFirst`）と同じ話を源にすることで「あらすじは最古話・credits は別の代表話」というソース不整合を避け、全話パースのコストも省く。
- その「1話目の説明文 1 本」自体は、**源優先マージ（`chooseDescription`）が選定した勝者**である。選定キーは ①構造（`<br>`）を持つもの優先 → ②源ランク `snapshot > nvapi > rss` → ③長さ、の順。抽出処理は源を意識しない（入力の出自に非依存）。源の選び方の詳細は [`dataflow.md`](dataflow.md) を参照。

## 3. 元データの実態（事実）

抽出方法を理解するには、まず**実際の説明文がどう書かれているか**を知る必要がある。以下はローカル各話コーパス＋ live snapshot 再取得の実測に基づく事実である。

### 3.1 構造化版とフラット版が混在する

説明文には HTML 上**2 つの書式**が存在する。

| 書式           | 由来             | 中身                                                                                   | 実測                               |
| -------------- | ---------------- | -------------------------------------------------------------------------------------- | ---------------------------------- |
| **構造化版**   | snapshot / nvapi | あらすじ・キャスト・スタッフ・© を `<br>`（多くは `<br><br>`）で区切る                 | `<br>` を持つ description が 99.5% |
| **フラット版** | チャンネル RSS   | 全要素を `<p class="nico-description">…</p>` 1 個に**区切り無しで連結**（`<br>` 0 個） | フラット長文は新着回転窓に限られる |

これが抽出可否を分ける**第一の事実**である。構造化版は要素の境界（`<br>` 由来の空行）が明示されるので分解できるが、フラット版は境界が消え、あらすじとキャストが区切り無しで直結する（例 `…シュプール:逢坂良太原作:佐賀崎しげる…`）。**よってフラット版は分解せず原文のまま扱う**（§5 ガード①）。

### 3.2 構造化版の中身は「役:値／役:値…」のブロック

構造化版を空行で段落に割ると、典型的には次の要素が並ぶ。

```
（あらすじのプロ―ズ）

役名:声優A／役名:声優B／役名:声優C …      ← cast ブロック
原作:○○／監督:△△／アニメーション制作:□□ …  ← staff ブロック
©○○製作委員会                            ← copyright
so12345678 ←前話 ／ 次話→ so12345679       ← 話リンク
```

ここに**第二の事実**がある。クレジット行は **全角 `／` でエントリを並べ、各エントリは `役:値`（コロン区切り）** で書かれる。コロンは半角 `:` が主だが全角 `：` も実在する。この「`／` 並び＋コロン区切り」が分解の足がかりになる。**そして本機能が欲しいのは右側の「値」（名前）だけで、左側の「役」（役名・役割ラベル）は捨てる。**

### 3.3 名前に付随するノイズ

値（名前）側には、発見に役立たない付随物が混ざる。これが掃除（§4）の対象になる。

| ノイズ                   | 例                                                                                              | 扱い                                                   |
| ------------------------ | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------ |
| 末尾の所属注記（括弧）   | `米山和仁（劇団ホチキス）` / `Aimer(DefSTAR RECORDS)` / 入れ子 `宮島礼吏（『誌名』（講談社））` | 末尾括弧を（入れ子対応で）除去                         |
| 作品名・曲名（引用符内） | `トロル「おしりたんてい」ポプラ社` / 主題歌 `『ププッとフムッと…』`                             | 引用符内を除去（曲名だけなら丸ごと落とす）             |
| 連結した複数名           | `高橋ナツコ、成田 順` / `奈須きのこ・TYPE-MOON`                                                 | 読点 `、,` で分割（中黒 `・` は分割しない＝§7）        |
| copyright のノイズ       | `©` 記号 / `©2017` 年 / `◯◯製作委員会`                                                          | 除去（制作実体だけ残す＝copyright マイニング）         |
| クレジットでないもの     | setlist 曲順 `01:曲名` / 役名・年                                                               | role が数値 → 不採用、役名は値でないので元から取らない |

## 4. 抽出方法（事実をどう実装に落としたか）

抽出は 2 段。①`parseDescription` が §3 の構造を段落分類し、cast/staff/copyright を構造として取り出す（synopsis 温存・lossless 込み）。②`extractCredits` がそこから**名前だけを掃除して 1 列に統合**する。

### 4.1 段落分類（`parseDescription`・§3.1/§3.2 に対応）

`isStructuredDescription` で `<br>` の有無を見て、無ければ分解せず全文 synopsis に温存（§5①）。ある場合は `stripHtml` で `<br>`/`</p>` を `\n` 化し、空行（`\n{2,}`）で段落に割り、各段落を**最初に一致した種類**で確定する: ① 話リンク/info → ② copyright（`©`/`製作委員会` 等のマーカー）→ ③ staff（`STAFF_KEYS` 38 語のいずれか＋コロン）→ ④ cast（staff キーワード無し・`／` で 2 件以上・各セグにコロン）→ ⑤ それ以外＝synopsis。staff を cast より先に判定するのが要点（両者は同じ `役:値` 形なので、staff キーワードの有無だけが判別子）。エントリ分解（`parseEntries`）は全角 `／` 分割＋最初のコロンで `役`/`値` に割り、1 セグメントでも割れなければブロックごと不採用（lossless）。

> この段で得る構造（cast=`{役,値}`、staff=`{役,値}`、studios、copyright 原文）は **§5 のガードを通った「構造的に妥当な行」だけ**。次段はここから名前を取り出すので、プロ―ズ・台詞・setlist の混入は既にこの段で弾かれている。

### 4.2 名前の掃除と統合（`extractCredits`・§3.3 に対応）

`parseDescription` の結果から、cast の値・staff の値・studios・copyright マイニング結果を集め、各値を `cleanValueToNames` で掃除して 1 列に統合・重複除去する。役（役名・役割ラベル）は**ここで完全に捨てる**（名前だけがタグ）。

`cleanValueToNames`（1 値 → 掃除済み名前配列）の順序と理由:

1. **作品名/曲名の除去**（`stripTitles`）: 引用符 `「」『』""` 内を削る。クレジット値内の引用符は常に作品名・曲名であって人物/社名ではない（`トロル「おしりたんてい」ポプラ社`→`トロル ポプラ社`、主題歌 `Aimer「曲名」`→`Aimer`）。曲名だけの値はここで空になり落ちる（＝主題歌の曲タイトルは発見に無価値なので捨てる）。
2. **末尾所属括弧の除去**（`stripAffiliation`）: 末尾が閉じ括弧なら対応する開き括弧まで（**深さを数えて入れ子対応**）落とす。`宮島礼吏（『誌名』（講談社））`→`宮島礼吏`。先頭にある括弧（`(株)カラー`）は名前の一部なので触らない。
3. **連結値の分割**（`splitNames`・safe モード）: 読点 `、，` と `,` で分割する（`高橋ナツコ、成田 順`→2 件）。**中黒 `・` では分割しない**（外国人名 `ロマン・トマ` 等を壊さないため＝§7 で測定して決定）。
4. **各片の再掃除＋ノイズ落とし**: 分割後の各片に末尾括弧が残れば再度 `stripAffiliation`。`isNoiseName` で**製作委員会名・引用符の残骸・年のみ・記号/数値のみ・文（句点を含むプロ―ズ）・極端に長い（30字超）**を落とす。

**copyright マイニング**（`mineCopyright`）: copyright 原文を `／ / 、 , ・` で分割し、各トークンから `©`/`(C)`/`®`、先頭の年（`©2016 …`）、`All Rights Reserved`、行頭の役ラベル（`原作:`）を剥がし、`stripTitles`・`stripAffiliation` を通して `isNoiseName` で篩う。`©カラー／EVA製作委員会`→`カラー`、`©藤本タツキ／集英社・MAPPA`→`藤本タツキ・集英社・MAPPA`、`©2016 とある製作委員会`→（空）。copyright だけは制作実体を取りこぼさないよう中黒 `・` も分割する（集英社・MAPPA を割る）。

統合後、順序を保ったまま重複除去して `credits: string[]` を返す。

## 5. 誤検知をどう防ぐか（precision の実現）

「構造的におかしい誤抽出ゼロ」を目指す仕組み（コードによる意味的保証ではなく、保守的なガードの積み重ね＋全数監査での実証。→ §9）。確実に解釈できないものは抽出しない。

| #   | 防ぐ誤検知                            | なぜ誤検知になるか                                          | どう防ぐか                                                                                       |
| --- | ------------------------------------- | ----------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| ①   | **フラット RSS の連結文字列**         | あらすじとキャストが区切り無しで直結し境界を一意化できない  | `<br>` 構造が無ければ**一切分解せず**全文 synopsis（`isStructuredDescription`）                  |
| ②   | **コロン 1 個だけの台詞・文**         | `セリフ:…` のような文を cast 化しうる                       | cast は **`／` で 2 件以上＋各セグにコロン**を必須化（`looksLikeCast`）                          |
| ③   | **各話要約・台詞コロンを含むプロ―ズ** | 「#3：突如…」のような段落が `役:値` に見える                | role/value に**句点 `。` を含めば不採用**（`hasProsePeriod`）＋**長さ上限**（値平均/最大・役長） |
| ④   | **ライブの曲順（setlist）**           | `01:曲名／02:曲名` の数値ラベルがクレジットに見える         | **role が数値・記号のみなら不採用**（`NUMERIC_ROLE_RE`）                                         |
| ⑤   | **一部だけ壊れたブロック**            | 1 行でも `役:値` に割れない混入があると残りだけ採るのは捏造 | `parseEntries` が**1 セグメント壊れたらブロックごと null**（lossless）                           |
| ⑥   | **主題歌の曲名・作品名**              | `主題歌:「曲名」` の曲名は 1 作品しか指さず発見に無価値     | `stripTitles` で引用符内を除去（曲名だけなら空で落ちる）。作詞/作曲/アーティスト人名は残す       |
| ⑦   | **製作委員会名・年・© 記号**          | `◯◯製作委員会`/`©2017` は単作品ゆえ無価値・名前でない       | `isNoiseName`／`mineCopyright` で除去                                                            |
| ⑧   | **役名・役割ラベル**                  | `衛宮士郎`（キャラ名）等は人物でなく単作品の役名            | `extractCredits` は**値だけ取り役を完全に捨てる**（構造的にラベルが出ない）                      |

## 6. 結果（カバレッジ実測）

**測定方法**: ローカル全シリーズの 1話目（`firstContentId`＝あらすじと同一ソース）を seriesId 昇順で系統抽出（n≈900）し、その**生 description を live snapshot（`filters[contentId]`・`<br>` 付き）で取り直して `extractCredits`** にかけた。**各話サンプルではなくシリーズ単位**（1 シリーズ = 1 票）。再現ハーネスは `scratch/measure-credits.mjs`（gitignore 配下・コミット対象外）。

**シリーズ単位（代表 n=897・2026-06-29 実測）**:

| 指標                      | 実数    | 率        |
| ------------------------- | ------- | --------- |
| **≥1 名が取れる（必達）** | **888** | **99.0%** |
| 取れなかった              | 9       | 1.0%      |

1 シリーズあたり平均 **21.2 名**（safe モード）。要件「7 割以上」を大きく満たす。旧設計の「演者/制作/studios 全部空＝7.5%」（≒92.5% カバレッジ）に対し、copyright マイニングと統合で **99.0%** まで上がった。

**種別別**（title/tags ヒューリスティック分類のため率は方向性。種別分類は本機能の一部でなく参考）:

| 種別   | n   | ≥1 名 |
| ------ | --- | ----- |
| アニメ | 637 | 98.9% |
| 映画   | 130 | 100%  |
| 舞台   | 60  | 98.3% |
| ライブ | 70  | 98.6% |

統合＋copyright マイニングにより、旧設計で薄かった舞台（旧 cast 31%）・ライブ（旧 cast 60%）も、制作実体・スタッフ・出演者のいずれかで ≥1 名がほぼ確実に取れるようになった。

## 7. 設計判断（測って決めたこと）

### 7.1 連結値の分割モード — **safe（読点 `、,` のみ）を採用**

`奈須きのこ・TYPE-MOON` や `高橋ナツコ、成田 順` のように、1 値に複数名が連結することがある。分割すれば recall が上がるが、外国人名の中黒（`ロマン・トマ`/`コミックス・ウェーブ・フィルム`）を割ると誤った名前を生む。3 モードを n=897 で実測した:

| モード           | 分割文字  | ≥1 名カバレッジ | 平均名数/シリーズ | 精度リスク                                                                               |
| ---------------- | --------- | --------------- | ----------------- | ---------------------------------------------------------------------------------------- |
| none             | 分割なし  | 99.0%           | 19.9              | なし                                                                                     |
| **safe（採用）** | `、 , `   | 99.0%           | 21.2              | なし（中黒は保護）                                                                       |
| all              | `、 , ・` | 99.0%           | 21.6              | **あり**: `・` 含む値 407 件中 122 件がカタカナ/英字のみ＝外国人名で、`all` は誤分割する |

**判断**: カバレッジは 3 モードとも 99.0% で同一。`all` の recall 上積みは平均 +0.4 名/シリーズと僅少な一方、外国人名 122 件を誤分割する precision 低下が確実。よって **safe を採用**（中黒 `・` は分割しない）。`奈須きのこ・TYPE-MOON` のような「中黒で繋いだ別実体」は 1 タグのまま残るが、誤った名前を作るより良いと判断。**copyright マイニングだけは例外で中黒も分割する**（`集英社・MAPPA` のように制作実体を取りこぼさないため。copyright 値に外国人名は出にくい）。

### 7.2 recurrence（再登場性）による取捨 — **ルールベースで実施・コーパス ≥2 フィルタは見送り**

発見に繋がらない「1 作品しかヒットしない名前」は落とす方針（§1.1）。これを次の**ルールベース**で実現している（§5⑥⑦⑧）: 曲名・作品名（`stripTitles`）、製作委員会名・年・© 記号（`isNoiseName`/`mineCopyright`）、役名・キャラ名（値だけ取り役を捨てる）。一方で作詞/作曲/編曲の**人名**や主題歌の**アーティスト名**は他作品で再登場するので残す。

**コーパス全体で出現 1 回の名前を facet から落とす ≥2 フィルタ**も検討し、測定した（safe + ≥2 フィルタ → カバレッジ 95.9%・要件は満たす）。しかし**見送った**。理由:

1. **サンプル頻度が当てにならない**: ユニーク名は代表 n=897 で 71.7% だが、これは全 6,601 シリーズの約 1/7 しか測っていないための過大評価で、全コーパスの真の出現頻度とは別物。サンプルで「1 回」でも全件では複数回のことが多い。
2. **詳細とフィルタの不整合**: 詳細は series JSON の credits、一覧は works.json の credits を読む。日次フル/毎時パーシャルの書き分けがあるため、頻度フィルタを両者に一貫適用するのは複雑で、片方だけだと「詳細に出るのにクリックすると 0 件」の死にチップを生む。
3. **無害**: 出現が稀な正しい名前をクリックしても、その 1 作品が返るだけで害はない。

将来、全コーパス頻度を一括計算できる文脈が整えば facet 限定の拡張として再検討しうる（本機能の必須要件ではない）。

## 8. 既知の制約（recall を捨てている所）

precision を守るため、以下は**意図的に取らない**（誤抽出はしないが、抽出もしない）。この欠落は (i) 注記でユーザーに開示している（§1.2）。

- **フラット RSS**: 構造判定①で丸ごと synopsis 行き。新着各話の暫定で、次 daily に snapshot 構造版へ置換されるので恒久的な欠落ではない。
- **2.5次元ミュージカルの `出演:キャラ 役：俳優` 形式**（舞台）: value が長く③で不採用になり、出演者を取れないことがある（制作・脚本は取れる）。誤った演者を出すよりよいと判断。
- **中黒で連結された別実体**（`奈須きのこ・TYPE-MOON`）: safe モードで 1 タグのまま（§7.1）。
- **役名・役割の区別**: credits は名前のみで、声優/監督/会社の区別は持たない（§1.5）。
- **credits の横断 facet**: 名前クリックの単発フィルタのみ。声優別・制作別のグローバルインデックスは未実装。

## 9. 検証基準

**この検証で「誤抽出ゼロ」が意味すること（射程）を最初に明示する**: 抽出した各名前が**構造的不変条件（プロ―ズ・台詞・曲順番号・引用符残骸・製作委員会名・年の混入なし）を破っていない**ことを確認した、という意味である。**正解データ（公式クレジット・人手ラベル）との 1 件 1 件の事実突合はしていない。** したがって「構造的におかしい誤抽出はゼロ」を保証する一方、「名前らしく見えるが事実は誤り」「役と人物の取り違え」「取りこぼし（recall）」は本検証の対象外である（§8）。

### 9.1 自動検証（テスト）

`tests/etl/description.test.mjs` で `parseDescription`（段落分類・lossless）と `extractCredits`（統合・名前のみ・所属括弧除去・safe 分割・copyright マイニング・主題歌曲名落とし・dedup・フラット/null）を網羅。`tests/web/detail.test.ts`（「演者/制作」1 行・dedup・(i)・`?credit=` リンク）／`tests/web/filter.test.ts`（credit フィルタ）でユーザー可視仕様を固定。全 543 テスト green。

### 9.2 人手スポットチェック（事実の正しさはこちらで補う）

不変条件が事実正しさを保証しない分は、**種別横断の代表サンプルを人手で目視**して補う。再現ハーネス `scratch/dump.mjs`（gitignore 配下）で live snapshot から 86 シリーズ（アニメ/映画/舞台/ライブ混在・既知作品含む）の 1話目を引き直してダンプ（`scratch/credit-dump.md`）。目視で構造的・事実的な誤抽出は確認されず（例: Fate/UBW で声優 17 名＋スタッフ＋ufotable＋主題歌アーティスト Aimer/Kalafina を正答、チェンソーマンで声優＋藤本タツキ・集英社・MAPPA、おしりたんていで声優＋トロル＋東映アニメーション。曲名・製作委員会名・年・役名は credits に出ていない）。

> **一文要約**: 「全数の構造的不変条件に対する違反ゼロ＋種別横断の代表サンプルの人手目視で誤りなし＋シリーズ単位カバレッジ 99.0% を live snapshot で実測。ただし全数の事実正解突合はしておらず、名前単位の事実誤り・recall 欠落は射程外。」
