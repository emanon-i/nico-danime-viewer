## 1. 機能概要

### 1.1 この機能は何か

**動画説明文からの自動タグ抽出機能**である。ニコニコの各シリーズの動画説明文（description）は、あらすじだけでなく**キャスト・スタッフ・制作会社・著作権表記**を 1 本の文字列に詰め込んで配信される。この機能は、その文字列を機械的に解析して

- **演者** — cast（声優名）
- **制作** — staff（スタッフ人名）＋ studios（制作会社名）
- **著作権** — copyright（© 行）

を**自動でタグとして抽出**し、作品詳細画面に表示する。さらに各人物タグをクリックすると、その人物が関わる作品の一覧へ絞り込める。

公式に構造化されたクレジット API は存在せず、説明文という非定型テキストが唯一の情報源であるため、**テキスト解析による自動抽出**という形をとる。自動抽出ゆえに必然的に不正確を含むので、本機能は「**間違ったタグを 1 件も出さない（precision 100%）**」を最優先の品質要件とし、確実に解釈できないものは抽出しない（→ §1.5・§5）。

### 1.2 ユーザーから見た振る舞い

機能の外形仕様（実装は `web/src/features/detail/detail.ts` `buildCredits` / `web/src/features/list/filter.ts`）:

| 振る舞い                          | 仕様                                                                                                                                                                                                    |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **詳細画面に演者/制作タグが出る** | 作品詳細に「演者」「制作」の 2 行を表示。演者＝声優名、制作＝スタッフ人名＋制作会社名のタグ列。**役名・役割ラベルは表示しない**（名前のみ）。                                                           |
| **空行は出さない**                | 演者・制作それぞれデータが無ければその行を出さない。両方空ならクレジット欄ごと非表示。                                                                                                                  |
| **クリックで一覧フィルタ**        | 各タグはクリックでその人物の作品一覧へ遷移。演者→ `?cast=<名前>`、制作→ `?staff=<名前>`。一覧は `work.cast` / `work.staff` にその名前を含む作品だけに絞り込む（完全一致）。既存のタグチップと同じ作法。 |
| **重複除去**                      | 同名は 1 つにまとめる。                                                                                                                                                                                 |
| **長い名前は省略**                | 20 文字超のチップは末尾省略＋全文ツールチップ。                                                                                                                                                         |
| **自動抽出の注記 (i)**            | 演者・制作それぞれの見出しに (i) ボタンを置き、ツールチップで「これらは説明文からの自動抽出のため、誤り・抜け・不正確が含まれる場合があります。」と明示する（`CREDIT_NOTE`）。                          |

この「不正確がありうる」注記は**仕様の一部**である。precision 100% を満たしても recall は犠牲にしている（取りこぼしがある）ため、欠落・不正確の可能性をユーザーに開示する。

### 1.3 用語定義

| 用語                     | 定義                                                                                                        |
| ------------------------ | ----------------------------------------------------------------------------------------------------------- |
| **説明文 / description** | 各話に紐づく動画説明文（1 本の文字列）。あらすじ・クレジット・©・話リンクを含む。                           |
| **構造化版**             | 要素を `<br>` で区切った説明文（snapshot / nvapi 由来）。分解できる。                                       |
| **フラット版**           | 全要素を区切り無しで連結した説明文（RSS 由来）。分解しない。                                                |
| **演者 / cast**          | 役名 → 声優の対。UI では声優名のみタグ化。                                                                  |
| **制作 / staff**         | 役割 → 人名/社名の対。UI では人名＋制作会社名をタグ化。                                                     |
| **studios**              | 制作会社。staff の制作系 role（`アニメーション制作`/`制作`/`製作`）から投影。                               |
| **copyright**            | © 行（原文保持）。                                                                                          |
| **synopsis**             | あらすじ等のプロ―ズ。抽出できなかった段落の温存先でもある。                                                 |
| **precision / recall**   | precision＝出した抽出のうち正しい割合（本機能は 100% 目標）。recall＝取りこぼさない割合（本機能は犠牲可）。 |
| **lossless（無損失）**   | 抽出に失敗しても原文の文字を取りこぼさず・捏造しないこと。                                                  |

### 1.4 入力 / 出力

- **入力**: 各シリーズの「1話目（最古話）」の説明文 1 本（→ §2）。
- **抽出処理**: ETL（`scripts/etl/description.mjs` `parseDescription`）が分類・抽出する（→ §4）。
- **出力（データ）**: SeriesEntry に `cast` / `staff` / `studios` / `copyright`（series 単位）。WorkEntry には人物フィルタ用に `cast` / `staff` の**名前配列のみ**。型定義の正本は [`data-inventory.md`](data-inventory.md) §2.6 / §4.1。
- **出力（画面）**: 詳細画面の演者/制作タグ列と、`?cast=`/`?staff=` による一覧フィルタ（→ §1.2）。

### 1.5 品質要件とスコープ外

**品質要件（precision 最優先）**:

- 出した演者/制作タグは **1 件残らず正しい**（誤抽出ゼロ・全数監査で実証。→ §8）。
- 確実に解釈できないものは抽出せず、原文（synopsis）に温存する（lossless）。
- recall（網羅率）は犠牲にしてよい。取れないものは取らない。

**この機能が扱わないこと**:

- **どの説明文を入力に選ぶか**（snapshot / nvapi / RSS のどれを採るか＝源優先マージ）は別トピック。本書は「選ばれた 1 本をどう分解するか」だけを扱い、源の選定は §2 で 1 段落だけ触れて [`dataflow.md`](dataflow.md) に委ねる。
- **演者/制作のグローバル facet 化**（声優別・制作別の横断インデックス）は未実装（人物クリックの単発フィルタのみ）。
- 役名・役割ラベルの表示（内部保持のみ・UI 非表示）。

## 2. 入力 — 何をパースするか

抽出処理（`parseDescription`）に渡すのは、**各シリーズの「1話目（最古話）」の説明文 1 本だけ**である。

- シリーズの全話をパースするのではなく、`chronoSort` 昇順で先頭＝最古話 1 件のみ。cast/staff はシリーズ内でほぼ一定なので 1話目で十分であり、あらすじ（`descriptionFirst`）と同じ話を源にすることで「あらすじは最古話・cast は別の代表話」というソース不整合を避け、全話パースのコストも省く。
- その「1話目の説明文 1 本」自体は、**源優先マージ（`chooseDescription`）が選定した勝者**である。選定キーは ①構造（`<br>`）を持つもの優先 → ②源ランク `snapshot > nvapi > rss` → ③長さ、の順。抽出処理は源を意識しない（入力の出自に非依存）。源の選び方の詳細は [`dataflow.md`](dataflow.md) を参照。

つまり本機能の抽出スコープは「源優先マージが選んだ 1 本」を入口とする。以降「説明文」はこの 1 本を指す。

## 3. 元データの実態（事実）

抽出方法を理解するには、まず**実際の説明文がどう書かれているか**を知る必要がある。以下はローカル 88k 各話コーパスの実測に基づく事実である。

### 3.1 構造化版とフラット版が混在する

説明文には HTML 上**2 つの書式**が存在する。

| 書式           | 由来             | 中身                                                                                   | 実測                                                |
| -------------- | ---------------- | -------------------------------------------------------------------------------------- | --------------------------------------------------- |
| **構造化版**   | snapshot / nvapi | あらすじ・キャスト・スタッフ・© を `<br>`（多くは `<br><br>`）で区切る                 | `\n` を持つ description が 99.5%（87,479 / 87,961） |
| **フラット版** | チャンネル RSS   | 全要素を `<p class="nico-description">…</p>` 1 個に**区切り無しで連結**（`<br>` 0 個） | フラット長文は 1 件のみ（＝新着回転窓）             |

これが抽出可否を分ける**第一の事実**である。構造化版は要素の境界（`<br>` 由来の空行）が明示されるので分解できるが、フラット版は境界が消える。フラット版では例えばあらすじとキャストが区切り無しで直結し、

```
…シュプール:逢坂良太原作:佐賀崎しげる…
```

のように「どこまでが台詞・あらすじで、どこからが声優名か」を一意に決められない。**よってフラット版は分解せず原文のまま扱う**（§5 ガード①）。

構造化版はほぼ全話で取得可能で、フラットに見えるのは「daily で構造版を再取得する前の新着各話」という時間的な回転窓に限られる。

### 3.2 構造化版の中身は「役:値／役:値…」のブロック

構造化版を空行で段落に割ると、典型的には次の要素が並ぶ。

```
（あらすじのプロ―ズ）

役名:声優A／役名:声優B／役名:声優C …      ← cast ブロック
原作:○○／監督:△△／アニメーション制作:□□ …  ← staff ブロック
©○○製作委員会                            ← copyright
so12345678 ←前話 ／ 次話→ so12345679       ← 話リンク
```

ここに**第二の事実**がある。クレジット行は **全角 `／` でエントリを並べ、各エントリは `役:値`（コロン区切り）** で書かれる。コロンは半角 `:` が主だが全角 `：` も 2.3% 実在する。この「`／` 並び＋コロン区切り」が分解の足がかりになる。

### 3.3 種別で書式に違いは「ある」

アニメ・映画・舞台・ライブで、**説明文の骨格（`<br>` 区切り＋あらすじ／クレジット／©）は共通だが、キャスト行の書き方に明確な違いがある**。これが種別ごとの cast 抽出率の差を生む。

| 種別                            | キャスト行の書式                                                                                                                          | 抽出可否                                                                                            |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **アニメ**                      | `役名:声優` をそのまま `／` 並び。例 `主人公:○○／ヒロイン:△△`                                                                             | きれいな `役:値` → **cast 取れる**                                                                  |
| **映画（劇場版）**              | アニメとほぼ同形                                                                                                                          | **cast 取れる**                                                                                     |
| **舞台（2.5次元ミュージカル）** | `出演:キャラ名 役：俳優名`。`出演:` の value 部に「キャラ名 役：俳優」が全角スペース込みで入る。例（薄桜鬼）`出演:相馬主計 役：梅津瑞樹…` | value が「キャラ＋役＋俳優」の長い塊になり `役:値` に割れない → **cast 取れない**（staff は取れる） |
| **ライブ**                      | 曲順（setlist）が `01:曲名／02:曲名…` の数値ラベルで並ぶ。アーティストは `役割：名前` 形で書かれる作品も一定数                            | 曲順は role が数値でクレジットでない。アーティストは取れる → **部分的**                             |

つまり「アニメ前提の `役名:声優` パターン」が成立するのはアニメ・映画で、舞台は出演者がキャラ名と抱き合わせの value になるため同じ規則では取れない。**種別専用パーサは作らず**、この違いは後述のガード（§5）で「曖昧なものは落とす」結果として自然に吸収される。

## 4. 抽出方法（事実をどう実装に落としたか）

§3 の事実から、抽出は次の 4 段で実装している（すべて `parseDescription`）。

### 4.1 構造判定（§3.1 の事実に対応）

まず `isStructuredDescription` で生 HTML に `<br>`（または実体参照 `&lt;br`）があるかを見る。**無ければ構造化されていない＝境界不明**なので、一切分解せず全文を synopsis に温存する（info 段落だけ除去）。`<br>` の有無で判定するのは、RSS フラット版も `<p>` ラッパ由来の改行 `\n` を持つため「`\n` の有無」では構造化版と区別できないからである（実測 200 件でフラット版の `<br>`=0）。

### 4.2 HTML 正規化と段落分割（§3.2 の「空行で要素が並ぶ」に対応）

`stripHtml` で `<br>` と `</p></div></li>`・各ブロック開始タグを `\n` 化し、残タグ除去・実体参照デコード・連続改行の畳み込みを行う。その後 `parseDescription` が **空行（`\n{2,}`）で段落配列に割る**。1 段落 = 1 要素（あらすじ / cast ブロック / staff ブロック / © / 話リンク）という §3.2 の構造を、そのまま段落の単位に落とす。

### 4.3 段落分類（最初に一致した種類で確定）

各段落を次の**優先順**で分類する。順序には理由がある。

1. **links / info** — `so… ←前話`／`次話→`／`動画投稿`等。話リンクは `so12345 :` のようにコロンを含み、放置すると後段で cast に誤判定されうるので**最初に除去**する（`extractEpisodeLinks` で prev/next/first を拾い、本文は synopsis に出さない）。
2. **copyright** — `©`/`Ⓒ`/`(C)`/`製作委員会`、または `原作／` 始まり。© 行は staff キーワードや `／` を含むことがあるため、staff 判定より**先に**マーカーで確定させる。
3. **staff** — `STAFF_KEYS`（`原作`〜`編曲` の 38 語）のいずれか＋コロンを含む段落。staff を **cast より先に**判定するのが要点。staff 行も「役:値／役:値」という cast と同じ形をしているので、cast を先に走らせると staff が cast に化ける。**staff キーワードの有無が両者の唯一の判別子**である。
4. **cast** — staff キーワードを含まず、`／` で 2 件以上に割れ、各セグメントにコロンを持つ段落（`looksLikeCast`）。§3.2 の「役名:声優を `／` で並べる」形そのもの。
5. **それ以外** — プロ―ズ＝synopsis。複数段落は `\n\n` 連結で保持。

### 4.4 エントリ分解（§3.2 の「`／` 並び＋コロン区切り」に対応）

staff / cast と判定した段落を `parseEntries` で構造化する。

- **全角 `／` で分割**する。`／` は §3.2 の通りクレジット行の正規の区切りだから。ただし分割は**ブロック内に限定**する。synopsis や © にも `／` が出る（例: `【各話概要】…とっとりアニメ編／…探訪！」編`）ため、説明文全体を `／` で割るのは禁止。
- 各セグメントを **最初のコロン**で `role` / `value` に割る（`/^([^：:]+)[：:]\s*(.+)$/`、半角 `:`・全角 `：` 両対応）。value 側にコロンが出ることがある（`収録曲:「…」作詞・作曲：…`）ため、最初のコロンで切るのが安全。
- value 内の社名・括弧注記（`(KADOKAWA)`／`（みなとそふと）`）・英字（`studioぴえろ`／`Lay-duce`／`MONACA`）は**値の一部として保持**。role 側の括弧（`蔵馬(南野秀一)`）・複合役（`監督・絵コンテ`）もそのまま role に保持する。**value は分割しない**: 1 セグメント = 1 文字列として `actors:[value]`／`names:[value]` のように単一要素配列に格納する（`・`/`、`/`,` で機械分割すると `演出:林祐一郎、朴性厚` のような複数人や `Wake Up,Girls！` のような固有名を壊すため、あえて分けず原文のまま持つ）。
- **1 セグメントでも `役:値` に割れなければブロックごと不採用**（`null` を返す）。半端な抽出を作らず、その段落は丸ごと synopsis に温存する（lossless。§5 ガード⑤）。
- staff のうち role が `アニメーション制作`／`制作`／`製作` のものは `studios` にも投影し、重複排除する（`isStudioRole`）。

cast の採用エントリは `{role, actors:[value]}`、staff は `{role, names:[value]}` として格納する。role（役名・役割）は保持するが UI では捨て、演者は声優名・制作は人名/制作会社名のみをタグ表示する（曖昧な役名の露出を避ける＝§1.2）。

## 5. 誤検知をどう防ぐか（precision 要件の実現）

§1.5 の品質要件「precision 100%・誤抽出ゼロ」を目指す仕組み（コードによる意味的保証ではなく、保守的なガードの積み重ね＋全数監査での実証。→ §8）。§3 で見た「誤検知を生みうる事実」ごとに、それを防ぐガードを置いている。確実に解釈できないものは抽出せず synopsis に温存する。

| #   | 防ぐ誤検知（どんな事実が誤検知を生むか） | なぜ誤検知になるか                                                                                      | どう防ぐか                                                                                                                                                                                                                 |
| --- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ①   | **フラット RSS の連結文字列**            | あらすじとキャストが `／` 無しで直結し（`シュプール:逢坂良太原作:佐賀崎しげる…`）、境界を一意化できない | `<br>` 構造が無ければ**一切分解せず**全文 synopsis（`isStructuredDescription` の安全弁）                                                                                                                                   |
| ②   | **コロン 1 個だけの台詞・文**            | `セリフ:…` のような文を cast 化しうる                                                                   | cast は **`／` で 2 件以上＋各セグにコロン**を必須化（`looksLikeCast`）                                                                                                                                                    |
| ③   | **各話要約・台詞コロンを含むプロ―ズ**    | 「#3：突如現れたメフィスト。…」のような要約段落が `役:値` に見える                                      | role/value に**文の句点 `。` を含めば不採用**（`hasProsePeriod`。『バクマン。』等は閉じ括弧が続くので除外）＋**長さ上限**（値平均 cast≤22/staff≤30・値最大 cast≤100/staff≤120・役≤50）。声優名・人名は短い、というのが根拠 |
| ④   | **ライブの曲順（setlist）**              | `01:曲名／02:曲名` の数値ラベルがクレジットの `役:値` に見える                                          | **role が数値・記号のみなら不採用**（`NUMERIC_ROLE_RE`）。正規の役名は数値のみにならないので損失ゼロ。value 側の数値風名（イラスト「029」/作曲「1869」）は正規なので許容                                                   |
| ⑤   | **一部だけ壊れたブロック**               | 1 行でも `役:値` に割れない混入があると、残りだけ採るのは捏造になる                                     | `parseEntries` が**1 セグメント壊れたらブロックごと null**。捨てた段落は synopsis に必ず残す（lossless）                                                                                                                   |
| ⑥   | **アニメ前提パターンの種別誤適用**       | 舞台『キャラ 役：俳優』形式・ライブ等にアニメ規則を当てると誤抽出                                       | **種別専用パーサを持たない**。②③で value が長い・形が違うものは自然に不採用になる（消極的不採用に統一）                                                                                                                    |

補足として、「確実にパース可」と「曖昧（保持にとどめる）」の切り分けは次の通り。

| 確実にパース可（採用）                    | 曖昧・誤検知リスク（非採用＝保持にとどめる）                           |
| ----------------------------------------- | ---------------------------------------------------------------------- |
| 構造版の段落境界（`\n\n`）＝ 99.5% で安定 | フラット RSS の連結文字列（`／` 無し直結あり）                         |
| staff キーワード＋コロンを持つ段落        | `／` のグローバル分割（synopsis/© にも `／` が出る）                   |
| `役:値／役:値…` を 2 件以上持つ段落       | value 内の `,`/`、` 強制分割（`Wake Up,Girls！` を割る誤り）           |
| © マーカー段落                            | role 側にコロンを含む稀例（first-colon で role 取り違え）→ cast 不採用 |

### 未分類の lossless 温存

分類できなかったクレジット様の段落は**破棄せず** synopsis 末尾に連結し、`unclassified` にも記録する（抽出に出した文字は必ず原文に存在＝捏造ゼロ。話リンク/info 段落のみ意図的に落とす＝§7）。`summarizeDescriptionParse` がパース時メトリクス（`{total, structured, flatFallback, withCast/Staff/Studios/Copyright, unclassifiedParagraphs, parsedRatePct}`）を出す。`parsedRatePct` は厳密には**構造化版の割合**（`structured / total`）であり、これと `unclassifiedParagraphs` の急変をフォーマットドリフトの検知に使う。

## 6. 種別ごとの結果

§3.3 の書式差は、そのまま種別ごとの抽出率に現れる。

- **アニメ・映画**: `役名:声優` がきれいに並ぶので演者がよく取れる。
- **舞台**: 制作は取れるが演者は薄い。ミュージカルの `出演:キャラ 役：俳優` 形式が③（value 長すぎ）で意図的に不採用になるため。
- **ライブ**: 曲順は④で落ち、アーティスト `役割：名前` が書かれる作品だけ部分的に取れる。

## 7. 既知の制約（recall を捨てている所）

precision を守るため、以下は**意図的に取らない**（誤抽出はしないが、抽出もしない）。この欠落は (i) 注記でユーザーに開示している（§1.2）。

- **フラット RSS**: 構造判定①で丸ごと synopsis 行き。新着各話の暫定で、次 daily に snapshot 構造版へ置換されるので恒久的な欠落ではない。
- **2.5次元ミュージカルの `キャラ 役：俳優` 形式**（舞台）: value が長く③で不採用。演者を取れないが、誤った演者を出すよりよいと判断。
- **役割ラベル（role）**: cast/staff とも role は内部保持のみで UI では捨て、名前だけ表示する。曖昧・不揃いな役名の露出を避けるため。
- **演者/制作の横断 facet**: 人物クリックの単発フィルタのみ。声優別・制作別のグローバルインデックスは未実装（§1.5 スコープ外）。

## 8. 検証基準

precision 100%・誤抽出ゼロ（§1.5）が満たされていることの実証。なお検証では**全 88k 各話をパースして監査**するが、これは「パーサがどの話でも誤抽出しない」ことの裏取りであり、**本番の抽出スコープは各シリーズ 1話目のみ**（§2）である点に注意（検証範囲 ≠ 本番抽出範囲）。

- **全 88k 各話の強条件全数監査**: cast 785,550＋staff 742,669 エントリを「role 非数値・プロ―ズ無し・時刻値無し・長さ妥当」で監査し**違反 0 件**。種別横断で実写舞台・実ライブからの誤抽出なし。
- **ライブサンプル監査**（2026-06-29 実測）: §6 の方法で 922 件をパースし、代表 598 シリーズで誤抽出 0。
- **捏造ゼロ**: 抽出した synopsis/cast/staff の文字は必ず原文に存在する（原文に無い文字を作らない）。クレジットの取りこぼしも synopsis 温存でゼロ。ただし話リンク/info 段落は意図的に落とすため、完全な文字単位 lossless ではない（§7）。
- **Exit Criteria（PH-0014）**: 構造化率 ≥ 99%（実測 99.5%）・誤抽出ゼロを全数で実証・`pnpm test`/`typecheck`/`lint`/`build` 通過。
