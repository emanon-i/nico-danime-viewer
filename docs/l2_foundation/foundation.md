## 1. 技術スタック

非サーバ・静的サイト前提。「予測どおりに動く・軽量」を満たす**最小構成**を選ぶ。

### 1.1 フロントエンド

- **Vite + TypeScript（バニラ、UI フレームワークなし）**
- **パッケージマネージャ＝pnpm**（依存導入・スクリプト実行は `pnpm`）。
- 選定理由: 静的サイトとして GitHub Pages にそのまま載る。React 等の重い実行時を持たず軽量・高速。
  作品一覧／ランキング／タグ・クールのフィルタ程度に大型フレームワークは過剰（over-engineering 回避）。
  TypeScript で `data/*.json` のスキーマを型として固定し、予測どおりの挙動を担保する。
- **却下案**: React/Next.js → 個人・静的・低複雑度には過剰。Astro → 候補だが当面 Vite で十分（将来再検討可）。
- **ダーク/ライトテーマ**: 既定は OS 追従（CSS `prefers-color-scheme`）、ユーザー選択（☀/🌙 トグル）は `localStorage` に保存。
  **保存するのはテーマ設定のみ＝個人化データではない**（視聴履歴等は扱わない方針と矛盾しない）。サーバ不要・即時反映。

### 1.2 バックエンド

- **専用バックエンドを持たない。** データ取得は Node.js（ESM, `.mjs`）の **fetch スクリプト**（`scripts/`）に集約。
- 選定理由: CORS と ToS（低頻度）の両面から、ブラウザ直 API は不可。取得を「ビルド/CI 時の前処理」に寄せ、
  実行時はただ静的 JSON を読むだけにする。Node 20+ のネイティブ `fetch` を使い依存を最小化。
- **取得元の役割分担（監査 `validation/api-availability.md` で確定）**:

| 用途                                                                                                           | 取得元                                                                                                     | 段階                         |
| -------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------- |
| 全作品カタログ／**五十音（読みの行 `col_key`）**                                                               | `list.json`（`title` / `col_key` / `url`）                                                                 | 非公式（静的JSON）           |
| **各話リスト・話順・各話再生数・支店判定**                                                                     | **nvapi `v2/series/<id>`**（`items[]`／`owner.channel.id=ch2632720`）＝主源                                | 非公式                       |
| タグ（dアニメキュレーションタグの正規化を含む・フラット1系統）・概要・再生数・投稿日・尺・各種カウンタ・サムネ | snapshot 検索 API（タグは各シリーズの最古話＝フル取得済み episode から導出）                               | **公式**                     |
| シリーズ概要                                                                                                   | **第1話の snapshot `description`** を流用（HTML 除去・「第1話のあらすじ」と明示）                          | 公式                         |
| **新着（最新の動画・毎時）**                                                                                   | **チャンネル RSS `ch.nicovideo.jp/ch2632720/video?rss=2.0`**                                               | 半公式                       |
| クール（年-季）                                                                                                | **タグ導出（`YYYY年<季>アニメ`）が主源**＋ `programlist.json`（今季）・period HTML（過去季）で補完（下記） | 公式（タグ）＋非公式（補完） |
| 勢いスコア                                                                                                     | **計算**（`0.5·delta_n + 0.3·velocity_n + 0.2·recency_n`。定義は下記／）                                   | 計算                         |

- **API の信頼度の階層**: 公式（snapshot v2・ドキュメントあり）＞ 半公式（チャンネル RSS）＞ 非公式（nvapi／静的JSON／period HTML）。
  **snapshot で取れるものは snapshot 優先**。**非公式に頼る箇所は変更検知アサートで守る**（§5.4）。
- **snapshot の制約**: `channelId` は**取得のみ可・filter 不可** → 取得後クライアント側で `channelId==2632720` 絞り込み。
  `contentId` は filter 可（`filters[contentId][0]=so…`）。`startTime` の filter は TZ（`+09:00`）必須。
- **タグの正規化（dアニメキュレーションタグ）**: snapshot `genre` は 99.95%「アニメ」一色で使えない。代わりに**第1話 `tags` の
  `dアニメ` マーカー付きキュレーションタグ**（**接尾 `…_dアニメ`/`…_dアニメストア`・接頭 `dアニメ_…` の両型**。例 `ドラマ/青春_dアニメ`・`dアニメ_音楽系`）を使う。
  **正規化**: マーカー（`_dアニメ(ストア)?$` または `^dアニメ_`）除去 → `/` 分割 → 各ラベル trim → 大小・全半角統一 → エイリアス吸収。素の `dアニメストア`（配信元）は除外。
  正規化したキュレーションタグは**通常タグと同じ平面のフラットなタグ**として扱う（別 facet にしない。品質が高いので候補の優先表示はしてよい）。
- **クール判定**: **主源＝タグ導出**。snapshot タグの `YYYY年<季>アニメ`（例「2022年秋アニメ」）をパースして放送季を導出する（追加 fetch 不要・放送季で正確・高 recall。複数季を含む場合は最古季を採用）。**補完**＝`programlist.json`（今季）→ period HTML（過去季・`anime.nicovideo.jp/period/<年>-<季>-danime.html`）で、タグから取れなかったシリーズの `cours` を埋める。period の series 紐付けは `/detail/<slug>` の**日本語タイトル正規化＋信頼度スコア**で結合（閾値以上のみ採用）。`startTime` 推定は使わない。取得は**変更検知アサート必須**（§5.4）。
  **実装状況**【実装済】: `scripts/etl/cours.mjs`（`deriveCoursFromTags` / `coursFromTags` / `mapCurrentCours` / `derivePastCours`・`parsePeriodHtml`・`matchPeriodEntriesToSeries`）・`scripts/fetch.mjs`（`runCoursPipeline` の3段直列）。過去季は `derivePastCours` が `COURS_FROM_YEAR`（既定 2016）以降の全季をループ。実測 約213季。
- **勢いスコアの定義**: **`hot_score = 0.5·delta_n + 0.3·velocity_n + 0.2·recency_n`**（各成分 0..1 に min-max 正規化）。
- **delta_n**：`delta_views = Σ(view_counter − prev_view_counter)` を min-max 正規化。`prev_view_counter` は**日次 fetch ごとに旧値を退避（1 スロットの bounded 保持）**し、差分を蓄積していく。
- **velocity_n**：`velocity = シリーズ合算再生数 ÷ max(1, 公開からの経過日数)` を `log1p` してから min-max 正規化。
- **recency_n**：`exp(−recency_days / tau)`（`tau=14` 日・`recency_days = now − 最新話 start_time`）。
- **正直な制約**: delta は退避が進むほど効く・解像度は日次（無制限の時系列は持たないため真の週/月トレンドではない）。画面の ⓘ ツールチップ「今の勢いの目安（正確な期間集計ではありません）」は維持。
- **カードの Hot 表示＝炎ティア（percentile ベース・ビルド時算出）**：🔥🔥🔥＝上位1% / 🔥🔥＝上位5% / 🔥＝上位10%（`design-system.md` §9.6.1・**現状は数値表示で炎ティアは設計・未実装**）。
- 実装は `scripts/etl/metrics.mjs`（`recalcSeriesMetrics`・set-based SQL）／スキーマ・UPSERT は 。

### 1.3 データストア

- **実行時（配信）のデータ層＝静的 JSON ファイル（`data/*.json`）。** 常時稼働 DB・サーバは持たない。
- **ビルド/CI 時の加工＝SQLite（ファイル DB・サーバ無し）**: 取得結果を SQLite に入れ、**DB 側で**増分 upsert・ランキング/勢い集計・タグ正規化（dアニメ 接頭/接尾）・フランチャイズ束ね（タイトル語幹＋`〜シリーズ` タグの union-find）・クール結合（タグ導出主源＋programlist/period 補完）を行い、用途別 JSON に export する。SQLite は**中間生成物**（再生成可能）。**状態（DB・`prev_view_counter`・HWM）の真実源は専用 artifact もしくは state ブランチ**、`actions/cache` は高速化フォールバック・状態書き込みは単一 state-writer の concurrency group（詳細は §7）。**スキーマ・インデックス・UPSERT/delta・PRAGMA・2 ジョブフローは **、全体のデータフローは [`dataflow.md`](dataflow.md)。
- **ユーザー状態（お気に入り/見た）＝ブラウザの localStorage**（「id＋フラグ」程度の小さな JSON・クライアント側・サーバに出さない・配信層とは別物）。大規模化する場合は IndexedDB を再検討。
- **export メタ**: 各 JSON に**最終更新時刻**を含める（全ページ固定フッターの「データ最終更新」表示用。詳細は [`dataflow.md`](dataflow.md)）。
- 増分取得の cursor（新着の最終 id、snapshot の最大 `startTime` 等）は SQLite／状態ファイルで保持（§2.5）。
- 選定理由: 読み取り専用配信は静的 JSON で十分。加工だけ SQLite を使い、常時稼働 DB・サーバは持たない。生成物はコミットせず再生成可能。

### 1.4 インフラ / デプロイ

- **GitHub Pages（配信）＋ GitHub Actions（定期 fetch & ビルド）**
- **pnpm 前提の Actions**: `pnpm/action-setup` ＋ `actions/setup-node`（`cache: pnpm`）で依存をキャッシュ。
- **cron は用途別に分割**:
- **新着用（毎時）**: チャンネル RSS から増分取得 → 新着系 JSON を更新。
- **発見用（日次）**: snapshot/カタログ/period から全件・タグ・クール・ランキング/勢いを再生成。
- 日次パスは `…/snapshot/version` の `last_modified` が前回から変わった時だけ走らせて無駄打ちを避ける（§2.5）。
- **却下案**: 自前サーバ/VPS → 常時稼働コストと運用負荷が見合わない。

## 2. アーキテクチャ

### 2.1 設計方針

- 採用パターン: **2 ステージ・パイプライン（Fetch/ETL → 静的 JSON）＋ データ駆動の静的フロント**。
  ドメイン（search / kana / tag / cours / ranking / new / series）単位で機能を分け、データアクセスは型付き JSON ローダ 1 箇所に集約。
- 選定理由: 「取得」と「表示」を物理的に分離（CORS・ToS・テスト容易性）。フロントは副作用が JSON 読み込みに限定され、
  予測可能・単体検証可能。外部依存（ニコニコ API）はスクリプト層に閉じ込め、フロントから隠蔽する（境界の明確化）。
- ファイル粒度: ドメイン単位でまとめる。インターフェース化は「外部依存の差し替えが要る箇所（= API クライアント）」のみ。
  単純な一覧・整形に UseCase 層やマッパーを増やさない（過剰分割の回避）。

### 2.4 画面構成（情報設計）

v1 は **3 画面**（**1. トップ**＝入口・発見／**2. 一覧**＝左フィルタ＋右グリッドの探索主戦場／**3. 作品(シリーズ)詳細**）。
状態は URL クエリで再現（共有・戻り位置が予測どおり）。各画面の ①ワイヤー（配置＋大きさ・モバイル/デスクトップ/ウルトラワイド）
②動線、使うデータ・REQ 対応、**ランキング方針**は別ファイル:

→ **[`screens.md`](screens.md)**（画面設計・アスキーワイヤー・ランキング方針）

**ランキング方針（要点）**: 支店スコープの**正確な週/月の急上昇（期間デルタ）は API から直接取れない**。**無制限の時系列蓄積はしない**が、
**`prev_view_counter`（1 スロット bounded）で前日比 delta を出し**、velocity・recency とブレンドして「勢い」とする（定義は §1.2 / ）。直接構築できるのは**累計再生数順・新着順・クール絞り**。
トップの2入口で役割分離: **「人気TOP」＝累計再生数順のみ（期間セレクタを置かない＝月間/週間の正確ランキングは作れないため）**、
**「Hot」＝勢い（前日比 delta＋velocity の目安）**。「できない期間」を UI に出さないことで予測どおりを保つ。

実行時はいずれも**事前生成した用途別 JSON を読むだけ**（API 直叩きなし）。

### 2.5 データパイプライン（増分＋定期フル）

> 詳細なデータフロー（源 → 取り込み → SQLite 加工 → 静的 JSON → 配信）と更新計画は [`dataflow.md`](dataflow.md) を正とする。ここは要点のみ。

- **新着（毎時・ハイウォーターマーク増分）**: チャンネル RSS の page1 を毎時取得し、保存済み cursor（最終 `guid`/watch id）より
  新しい item だけ追加（初回は `?page=` を遡ってシード）。境界は `>=`＋**id dedup**（遅延・同時刻の取りこぼし/重複を防ぐ）。
- **発見系の差分（日次）**: snapshot は `filters[startTime][gte]=<保存済みの最大 startTime>`（TZ 付き）で日次差分を取得。境界 `>=`＋id dedup。
- **可変メトリクスは日次フルリフレッシュ**: `viewCounter` 等は HWM 増分では拾えない（既存話の数値が伸びる）→
  ランキング/勢い用に**日次で全件を再取得**（`_offset` 上限 100000 は `startTime` 範囲分割で回避）。
- **version ゲート**: 日次フルの前に `…/api/v2/snapshot/version` の `last_modified` を確認し、**前回から変わった時だけ**走らせる（無駄打ち回避）。
- 増分の cursor は `data/state/`（git 管理外）に保持。

## 3. 非機能要求

- **挙動の一貫性**: ランキング・並び順・フィルタは決定的（同じ操作→同じ結果）。隠れた重み付け・自動再生を実装しない。
- **API マナー**: UA 必須・**前回レスポンス時間と同じだけ待つ**・全件取得はページング/期間ウィンドウで分割。取得はスクリプト層のみ。
- **パフォーマンス**: 実行時は静的 JSON 読み込みのみ。約8.7万件（各話）規模のため、フロントは
  **用途別に分割・軽量化した JSON**（例: ランキング上位、シリーズ集約）を読む。<!-- TODO: 分割粒度はL3で確定 -->
- **更新頻度方針**（実証で確定）:
- **新着（最新の動画）= 毎時**。源は**チャンネル RSS**（`ch.nicovideo.jp/ch2632720/video?rss=2.0`・チャンネル直＝支店絞り不要・pubDate が日中更新）。
- **発見系（全件/ランキング/カタログ/タグ/クール）= 日次**。snapshot 索引は毎日 AM5:00 頃更新のため日次で十分。
- GitHub Actions の cron を**新着用（毎時）＋発見用（日次）に分割**（§1.4・§2.5）。
- **変更検知（壊れたら気づける）**: 上流の仕様・構造変化を**取得時に検知して失敗させる**（詳細は §5.4）。
  必須フィールド欠落・件数の急減/0・HTML 構造変化・空レスポンスを検知したら **Actions を fail** させ、
  **壊れた/空の JSON を公開しない**（前回正常な公開物を保持）。サイレント破損を許さない。非公式源（RSS/period/静的JSON/nvapi）は特に厳守。
- **セキュリティ**: 公開静的サイトの軽い脅威モデルと対策は [`security.md`](security.md)。要点＝外部文字列はエスケープ・概要 HTML は allowlist サニタイズ（`innerHTML` 禁止）、外部リンクは `noopener noreferrer`＋URL allowlist、CSP（`<meta>`）、ブラウザは自 JSON ＋画像 CDN のみ取得（API/RSS は直叩きしない）、Actions は最小権限・依存固定、個人データは localStorage のみ（クライアント側）。

### 5.4 取得時の変更検知（上流仕様変化への防御）

### 6.1 結論（想定どおり作れそうか）

- **v1 スコープ（全作品ブラウズ・再生数ランキング・タグ・クール・五十音・新着・deep-link）は実データで取得可能＝実現可能。**
  支店規模 `totalCount=87,327`、`channelId==2632720` のクライアント側絞り込み、`viewCounter`/`startTime`/`tags` 等の取得、
  `_sort` と `filters[startTime]` の動作、補助源（list.json 6,698 件 / programlist.json 75 件）、チャンネル RSS の毎時新着、period のクールをいずれも実証。
- **設計に織り込む前提（実証で判明）**:

1.  **`channelId` は filter 不可** → 取得後クライアント側で `==2632720` 絞り込み。`contentId` は filter 可。
2.  **`startTime` filter は TZ 必須**（`+09:00` 無しは HTTP 400）。全件取得の期間ウィンドウ分割もこれで行う。
3.  **`genre` が粗い（実測は一律「アニメ」）** → **タグは第1話 `tags` の `dアニメ` キュレーションタグ（接頭/接尾両型）を正規化してフラットに取り込む**（別 facet 化しない）。
4.  **新着の毎時源＝チャンネル RSS**。発見系は snapshot 日次。
5.  **クールはタグ導出（`YYYY年<季>アニメ`）を主源**に（過去季可・追加 fetch 不要）＋ programlist（今季）/period HTML（過去季）で補完。`programlist.json` の画像キーは **`imgpagh`**。
6.  **関連シリーズ/続編の関係フィールドは無い** → 共有フランチャイズタグでベストエフォート束ね。

- **将来機能は API 単独では不可**: 正確な期間デルタ（急上昇）や視聴履歴を使う個人化は
  **視聴履歴を集める別プロジェクト（ブラウザ拡張等）に依存**し本ビューアのスコープ外。L1 の優先度と整合。
