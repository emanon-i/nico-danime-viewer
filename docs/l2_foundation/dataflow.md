# データフロー＆更新計画（L2）

> `foundation.md` の補足。**データの源 → 取り込み → 加工(SQLite) → 静的JSON → 配信** の流れと、
> **低頻度（ニコニコ/支店に迷惑をかけない）と鮮度（新着は毎時）を両立する更新計画**を定める。
> 実証の裏付けは `validation/api-availability.md`。コードは書かない（L3 / gen-code）。

## 1. アーキテクチャ層（サーバ・常時稼働 DB は持たない）

```
 ┌─ 源（外部） ────────────────────────────────────────────────┐
 │ snapshot 検索API v2(公式)  ch RSS ch2632720/video(半公式)     │
 │ period HTML / list.json / programlist.json / nvapi series(非公式)│
 └───────────────┬───────────────────────────────────────────┘
                 │ 取り込み（GitHub Actions 内・サーバ無し）
                 │  ・HWM 増分（RSS=guid毎時／snapshot=startTime>= 日次差分。境界>=＋id dedup）
                 │  ・日次フルリフレッシュ（viewCounter 等の可変メトリクス用）
                 ▼
 ┌─ 加工: SQLite（ファイルDB・Actions 内・サーバ無し）──────────┐
 │ upsert（増分マージ）／ランキング・勢い(=合算再生数÷経過日数)集計 │
 │ タグ正規化(dアニメ 接頭/接尾)／フランチャイズ束ね／period クール結合│
 └───────────────┬───────────────────────────────────────────┘
                 │ export（用途別の静的 JSON）
                 ▼
        data/*.json ──► GitHub Pages（静的配信。ブラウザからDB直クエリはしない）
                 │
                 ▼
        ブラウザ：JSON を読むだけ ＋ ユーザー状態は IndexedDB（クライアント側・サーバに出さない）
```

- **ブラウザは静的 JSON を読むだけ**。SQLite はビルド/CI 時の**加工用**で、配信物には出さない（JSON に export する）。
- **ユーザー状態（お気に入り／見た）は IndexedDB**（クライアント側のみ・サーバへ送らない）。配信層とは別物。

## 2. データ源（信頼度の段階）

| 源 | 役割 | 段階 |
|----|------|------|
| snapshot 検索API v2 | 全件メタ・タグ・再生数・投稿日・第1話タグ（contentId 直引き） | **公式** |
| チャンネル RSS `ch.nicovideo.jp/ch2632720/video?rss=2.0` | **新着（最新の動画・毎時）** | 半公式 |
| period HTML / list.json / programlist.json / nvapi `v2/series/<id>` | クール・カタログ/五十音・各話リスト | 非公式 |

- **snapshot で取れるものは snapshot 優先**。非公式源は**変更検知アサートで守る**（`foundation.md` §5.4）。

## 3. 取り込み方針（HWM 増分＋日次フル）

- **新着（毎時・HWM 増分）**: RSS page1 のみ取得し、保存済み cursor（最終 `guid`/watch id）より新しい item だけ採用。境界 `>=`＋**id dedup**（遅延・同時刻の取りこぼし/重複対策）。初回のみ `?page=` 遡及でシード。
- **発見系の差分（日次・HWM）**: snapshot を `filters[startTime][gte]=<保存済み最大 startTime>`（TZ 付き）で日次差分。境界 `>=`＋id dedup。
- **可変メトリクスは日次フルリフレッシュ**: `viewCounter` 等は既存話の数値が伸びるため HWM では拾えない → ランキング/勢い用に**日次で全件再取得**（`_offset` 上限 100000 は `startTime` 範囲分割で回避）。
- 増分 cursor は SQLite（または `data/state/`）に保持。

## 4. 加工: SQLite（ファイル DB・Actions 内・サーバ無し）

- 取得結果を SQLite に入れ、**DB 側で**: 増分 upsert ／ ランキング・勢い（合算再生数 ÷ 公開からの経過日数）集計 ／
  タグ正規化（dアニメ 接頭 `^dアニメ_`・接尾 `_dアニメ(ストア)?$` の両対応）／ フランチャイズ束ね（共有タグ）／ period 由来クール結合。
- **SQLite の置き場**:
  - **第一候補＝GitHub Actions のキャッシュ**（key 付き・`git` を膨らませない・壊れても再生成可）。
  - 必要なら **repo にコミット**も可（履歴で差分追跡したい場合）。いずれも**再生成可能**な中間生成物として扱う。
- DB は**中間生成物**。配信には出さず、§5 の用途別 JSON に export する。

## 5. 出力（用途別の静的 JSON）

- `data/*.json`（`works / ranking / tags / cours / kana / new / series` 等）に export → **GitHub Pages で静的配信**。
- **トップのタグ導線**: Hotのタグ＝Hot（勢い）上位 N 作品のタグを集計した頻出タグ、人気のタグ＝人気TOP（累計）上位 N 作品の頻出タグ、ランダムタグ＝正規化タグ辞書からサンプル。いずれも**正規化済みタグ**から生成して JSON に出力（click → 一覧 `?tag=…`）。
- **export メタ**: 各 export に**最終更新時刻（取得/ビルドのタイムスタンプ）**を含める（ヘッダ About モーダルの「データ最終更新」表示用。新着=毎時・発見系=日次のため代表値を持つ）。
- フロントは**事前生成 JSON を読むだけ**（API/DB を直叩きしない）。分割粒度は L3。

## 6. ユーザー状態（IndexedDB・クライアント側）

- **お気に入り／見たマーク**は **IndexedDB**（ブラウザ内）に保存。**サーバへ送らない・配信層と別物**。
- 将来、視聴履歴拡張（別プロジェクト）が入っていれば「見た」を自動反映できる（連携は別プロジェクト側の責務）。

## 7. 更新計画（低頻度と鮮度の両立）

**方針: リクエスト数を最小化（条件付きGET・version ゲート・差分・並列禁止・レート遵守）しつつ、鮮度は新着＝毎時で確保。発見系の鮮度は日次で十分。**

| 対象 | 頻度 | やり方（負荷最小） |
|------|------|------------------|
| **新着（RSS）** | **毎時** | **page1 のみ**（数KB）＋**条件付き GET（`If-Modified-Since`/`ETag` → 304 なら本文無し＝ほぼ無負荷）**。HWM で dedup し増分のみ採用。初回のみ page 遡及でシード |
| **発見系フル（snapshot/カタログ/period）** | **日次** | **version ゲート**（`…/snapshot/version` の `last_modified` が変わった時だけ全件パス）。**AM5:00(JST) 索引切替後の閑散帯に1回**。**逐次取得（並列で叩かない）**・**前回レスポンス時間ぶん待機**・`_offset` 上限は `startTime` 範囲分割・**503 は 5 分以上バックオフ** |
| **各話（nvapi series）** | **低頻度/オンデマンド** | 全件を毎回引かない。**RSS で新着が出た／変化したシリーズだけ**、または**週次程度**に更新 |

- **各源のフェイル時**: 変更検知アサート（`foundation.md` §5.4）が外れたら **Actions を fail** させ、**壊れた/空の JSON を公開しない**（前回正常な公開物を保持）。
- ToS 厳守（非営利・User-Agent 必須・前回レスポンス時間ぶん待機・503 バックオフ）。
