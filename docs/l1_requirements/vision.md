# dアニメストア ニコニコ支店ビューア 要件・意思決定ドキュメント

> L1（要件・意思決定）。L0（`docs/l0_ideas/requirement.md`）の要求を構造化したもの。
> 技術詳細は L2、機能の受け入れ条件は L3 で確定する。

## 1. プロダクトの意思と背景

### 1.1 解決したい課題・ペイン

dアニメストア ニコニコ支店（docomo 本店とは別サービス＝ラインナップもプレイヤーも別）の公式 UI は見づらく、
「観たい・観るべき作品に出会う」までの導線が弱い。視聴可能な作品があっても発見されないまま埋もれる。
ニコニコの公開 API でメタデータ（再生数・タグ・ジャンル・投稿時間）は取得できるため、
見やすい発見サイトを軽量な静的サイトとして作る。

### 1.2 想定ユーザー／ペルソナ

- ニコニコ支店の加入者（または検討者）で、「何を観るか」をその場で発見したいライト〜ミドル層。
- ランキング・タグ・ジャンル・クールからザッピング的に作品を探したいユーザー。
- スマホ／PC ブラウザの両方で軽快に閲覧したいユーザー。

### 1.3 ビジネス上のゴール

- 個人による非営利・非公式プロジェクト。収益化しない。
- 支店ラインナップの発見性を高め、視聴は公式プレイヤーへ送客（deep-link）して公式体験を補完する。
- 公開 API のマナー（User-Agent 必須・低頻度・非配布）を守り、持続可能に運用する。

### 1.4 このプロダクトで大事にすること

- **予測どおりに動く**：同じ操作で同じ結果。ランキングや並び替えに隠れた重み付けをしない。
- **時間を奪わない**：自動再生・無限スクロールの罠・ダークパターンをしない。素早く目的に到達できるようにする。
- **迷わせない**：探し方（ランキング/タグ/クール）を分かりやすく分ける。
- **見やすさの理由を説明できる**：思いつきでなく、上記に沿っているかで判断する。

## 2. 制約条件・前提

### 2.1 動作環境

- 静的サイト（ブラウザ。スマホ／PC 両対応）。サーバサイドの常時稼働を持たない（GitHub Pages 配信）。

### 2.2 法規制・コンプライアンス

- 公開・非営利利用が前提。snapshot 検索 API は **User-Agent 必須・低頻度**。
- 作品データのキャッシュ／取得結果を**再配布しない**（生成 JSON は git 管理外）。
- 動画・字幕等の本体は扱わない。視聴は公式プレイヤーへの **deep-link のみ**（支店本編は有料チャンネル動画）。

### 2.3 外部システム連携

- カタログ主軸: `site.nicovideo.jp/danime/static/data/list.json`（全作品）、`programlist.json`（今季）、
  `anime.nicovideo.jp/period/<年>-<季>-danime.html`（クール・過去季）。
- メタデータ: ニコニコ snapshot 検索 API
  `https://snapshot.search.nicovideo.jp/api/v2/snapshot/video/contents/search`
  - `q=dアニメストア` / `targets=tagsExact` → レスポンスの `channelId == 2632720`（支店）でクライアント側絞り込み
    （`channelId` は filter 不可・取得のみ可）。
- ブラウザ直 API は CORS 不可 → ローカル/CI の fetch スクリプトで静的 JSON 化してから静的サイトが読む。

## 3. 要求と意思決定

### v1（今回作る範囲）

#### 全作品ブラウズ

- **ID**: REQ-0001 / **優先度**: 高
- **背景・動機**: 支店ラインナップ全体を一望できる入口が薄く作品が埋もれている。まず「全作品を見やすく並べる」基盤があることで、ランキング・タグ・ジャンル・クールの発見軸の土台になる。網羅性自体がユーザーの安心感と回遊の起点になるため最優先。

全作品を一覧・スクロール／検索・快適にブラウズできる画面。カタログ（list.json）＋ snapshot API で構築。
各話リスト・話順は **nvapi v2 series**（支店判定 `owner.channel.id=ch2632720`）を主源とし、シリーズ概要は**第1話の description を流用**（HTML 除去・「第1話のあらすじ」と明示）。

#### 人気ランキング（累計＋勢いスコア）

- **ID**: REQ-0002 / **優先度**: 高
- **背景・動機**: 「人気＝とりあえずの安全な選択肢」を求める行動は強く、発見の中核。snapshot API から `viewCounter`（累計）が取れ、低コストで“人気順”を予測どおりに提供できる。迷ったユーザーに最初の一歩を与える。

2 系統を提供し、トップでは 2 入口に割り当てる:
- **累計再生数順（「人気TOP」）**: `viewCounter` の降順（直接構築可）。**期間セレクタは置かない**（履歴非蓄積で月間/週間が作れないため、全期間累計のみ）。
- **勢い順（「Hot」・蓄積なし近似）**: `viewCounter` ÷ 投稿からの経過日数 等の**近似スコア**で「いま勢いがある」感を出す（真の期間デルタではない＝「勢い（目安）」として扱う。具体 formula は L3）。

> 検証結論（`docs/l2_foundation/validation/api-availability.md` 2回目）: 支店スコープの正確な週/月/急上昇デルタは API から直接取れず、
> かつ**自前の `viewCounter` 時系列蓄積は採用しない**ため、勢いは蓄積なし近似で代替する。

#### タグ別ブラウズ（フラット。dアニメキュレーションタグの正規化を含む）

- **ID**: REQ-0003 / **優先度**: 中
- **背景・動機**: 「気分・好みから探したい」探索行動に応える軸。`genre` 欄は 99.95%「アニメ」一色で使えないが、**監査で第1話の `tags` に `dアニメ` マーカー付きのキュレーションタグ**（接尾 `…_dアニメ`/`…_dアニメストア`・接頭 `dアニメ_…` の両型。例 `ドラマ/青春`・`sf/ファンタジー`・`音楽系`）があり、品質の高いサブジャンル的タグとして使えると確認（`validation/api-availability.md`）。これを正規化し、**通常タグと同じ平面のフラットなタグ**として扱う。

一覧サイドバーの**「タグ」フィルタ（1系統・フラット）**。dアニメキュレーションタグは**正規化**して取り込む（`dアニメ` マーカー＝`_dアニメ(ストア)?$` または `^dアニメ_` を除去 → `/` 分割 → 大小・全半角統一 → エイリアス吸収。接頭/接尾の両対応）。品質が高いので候補の優先表示はしてよいが、**別 facet（カテゴリ等）には分けない**。`genre` 欄は使わない。自由タグの絞り込み（REQ-0008）と同一系統。

#### 五十音ナビ（あ〜わ）

- **ID**: REQ-0011 / **優先度**: 中
- **背景・動機**: 作品数が多く、目的のタイトルへ素早く辿り着く「あいうえお順」の入口が要る。**監査で list.json の `col_key`（読みの五十音「行」）が全6,698件・欠落0で取得可能**と確認。これで五十音ボタン（あ〜わ）の行振り分けが予測どおり作れる。

一覧ページの五十音ボタンで `col_key` の行に絞り込む。並び替えにも「五十音順」を用意（同じ行順・行内はタイトル文字列順フォールバック）。**制約**: 完全な読み（yomi）は無いため**行内の厳密な50音順ソートは不可**＝行で括るのは正確、行内はタイトル文字列順フォールバック。

#### クール（季節）別ブラウズ

- **ID**: REQ-0004 / **優先度**: 中
- **背景・動機**: アニメ視聴者は「今季」「あの頃」という季節軸での想起・探索が強い。`startTime` および補助データ（period／programlist）でクール単位に束ね、時間軸の発見導線を提供する。話題性と懐古の両方を拾う。

放送クール単位のブラウズ。**監査の結果、`startTime` でクール判定できるのは現行季のみ**（過去作はバルク投稿日で放送季と無関係）。
今季は programlist、過去季は period ページを補助に。判定できない作品は「クール不明」で正直に扱う。

#### タグ・ブラウズ

- **ID**: REQ-0008 / **優先度**: 中
- **背景・動機**: 作品メタには豊富なタグが付き、ジャンルより細かい嗜好の手がかりになる。タグで辿れることで、ユーザー自身が探索軸を選べる能動的な発見を支える。

タグによる絞り込み・一覧（ユーザーが選んだタグで作品を探す）。

#### 新着一覧（最新の動画が軽く並ぶ）

- **ID**: REQ-0010 / **優先度**: 中
- **背景・動機**: 「今 追加された最新話」を軽く眺めたい欲求がある。トップなどに最新の動画が並ぶ枠があると、定期的に覗く動機になる。**監査で支店チャンネル RSS（`ch.nicovideo.jp/ch2632720/video?rss=2.0`）が日中更新（毎時級）と確認**できたため、新着は毎時で出せる（snapshot 索引は日次なので発見系と源を分ける）。

最新の動画が軽く並ぶ枠（トップのセクション＋一覧の新着順状態）。**新着＝チャンネル RSS で毎時更新**、発見系（全件/ランキング/タグ/クール）＝snapshot 日次。源・cron 分割は L2 §1.4 / §2.5 / §3。
なお新着は **新着シリーズ（シリーズ初出）と最新話（話単位）の 2 系統**に割れる（画面/データ詳細は L2 §2.4 / screens.md）。

#### 公式プレイヤーへの deep-link

- **ID**: REQ-0005 / **優先度**: 高
- **背景・動機**: 本ツールは発見に特化し、視聴は公式に委ねる方針。各作品から公式プレイヤーへ直接遷移できることで「見つけたらすぐ観られる」体験を完結させつつ、本体を扱わない非配布原則を守れる。発見から視聴への摩擦を最小化する。

`contentId` / series id から公式プレイヤー（`nicovideo.jp/watch/<id>`・`/series/<id>`）へ deep-link。

#### クイックアクセス入口（トップ）

- **ID**: REQ-0013 / **優先度**: 中
- **背景・動機**: 広いカタログでは「とりあえずどこから見るか」の入口が要る。トップに **[今期][新着][Hot][人気TOP]** のボタン列を置き、各々が**一覧の特定プリセット状態**（クール=現行季×勢い／新着シリーズ順／勢い順／累計順）を開く。予測どおりで迷わせない発見の起点。

トップのクイックアクセス＝既存 REQ（0002/0004/0010）へのプリセット導線。プリセット定義は L2 screens.md の表に従う。

#### ダーク/ライトテーマ切替

- **ID**: REQ-0012 / **優先度**: 中
- **背景・動機**: 長時間ブラウズする道具として目に優しい表示を選べると良い。既定は OS 設定（`prefers-color-scheme`）に追従し、ユーザーが ☀/🌙 トグルで上書きした選択は `localStorage` に保存する。**保存するのはテーマ設定のみ＝個人化データ（視聴履歴等）ではない**ため、ユーザー主権・非個人化の方針と矛盾しない。

ヘッダのトグルでダーク/ライトを切替（即時反映・サーバ不要）。

#### 関連シリーズ/続編への導線（ベストエフォート）

- **ID**: REQ-0014 / **優先度**: 中
- **背景・動機**: 「2期/続編/前作」へ直接飛べると回遊が締まる。ただし**監査の結果、関係を返す公式フィールドや作品グルーピング ID は存在しない**（nvapi・静的JSON・Nアニメ作品ページのいずれも `related` は空）（`validation/api-availability.md`）。代わりに**共有フランチャイズタグ**（作品名タグ／`〜シリーズ` タグ）で束ねれば、取れた範囲で関連を出せる。

作品詳細に「関連シリーズ/続編（2期等）」セクションを**ベストエフォート**で表示。共有フランチャイズタグの正規化＋少数の手動補正で束ね、タイトル語幹は弱い補助。**取れない作品は非表示**（無理に推定しない）。

#### お気に入り／見たマーク（クライアント保存）

- **ID**: REQ-0015 / **優先度**: 中
- **背景・動機**: 「気になる作品を後で」「もう見た」を手元で管理できると回遊が締まる。サーバを持たない方針なので、状態は**ブラウザの IndexedDB（クライアント側のみ・サーバに出さない）**に保存する。視聴履歴拡張（別プロジェクト）が将来入れば「見た」を自動反映できる（連携は別プロジェクト側）。

一覧カード（左上）と作品詳細に **♥ お気に入り／✓ 見た** トグル。一覧で「♥お気に入りだけ／✓未視聴だけ」の絞り込み（任意・軽量）。状態は IndexedDB。**個人化データはサーバへ送らない。**

### 将来（v1 スコープ外）

#### 急上昇（正確な期間デルタ）

- **ID**: REQ-0006 / **優先度**: 低（スコープ外）
- **背景・動機**: 「今 伸びているもの」を**正確に**出すには `viewCounter` の時系列差分（伸び率）が要る。だが検証の結論として、支店スコープでこれを直接取れる API は無く、**自前の時系列蓄積は保守・容量コストから採用しない**。本ビューアでは**蓄積なしの近似「勢いスコア」（REQ-0002）で代替**する。

正確な期間デルタ／伸び率の急上昇、および「自分のアーリー度」系は、**視聴履歴を集める別プロジェクト（ブラウザ拡張）の領分**＝本ビューアの対象外。近似の「勢い」は REQ-0002 でカバー。

#### 近い作品の提示（リコメンド）

- **ID**: REQ-0009 / **優先度**: 低（将来）
- **背景・動機**: 「これが好きなら次はこれ」を出せれば発見価値が高い。ただし根拠を説明できる（タグ・ジャンル類似など透明な）形に限る。v1 のブラウズ軸を固めてから検討する。

タグ／ジャンル類似にもとづく、根拠の見える「近い作品」提示。**v1 では作らない。**

#### 視聴履歴を使った個人化

- **ID**: REQ-0007 / **優先度**: 低（別プロジェクト依存・スコープ外）
- **背景・動機**: 「続きから／全話視聴済み」などは便利だが、**視聴履歴を集める別プロジェクト（視聴履歴ブラウザ拡張など）に依存する**。本ビューア単独では実現できず、その別プロジェクトが出来てからの話。

視聴履歴にもとづくおすすめ・続きから等。**本ビューアの対象外（別プロジェクトの産物に依存）。**

## 4. 非機能要求（高レベル）

- **挙動の一貫性**: ランキング・並び順・フィルタは「同じ操作で同じ結果」。隠れた重み付け・自動再生をしない。
- **API マナー**: User-Agent 必須・低頻度。取得は fetch スクリプト側に集約。
- **配信形態**: 静的サイト（CDN/静的ホスティングで配信可能なこと）。
- **パフォーマンス**: スマホでも軽快に一覧表示。<!-- TODO: 作品数規模・初期表示目標はL2で確定 -->
- **更新頻度**: **新着（最新の動画）はチャンネル RSS で毎時**、発見系（全件/ランキング/タグ/クール/カタログ）は **snapshot 日次**。源と cron を分ける（L2 §1.4 / §2.5 / §3）。
- **配信停止作品の扱い**: 支店ラインナップには配信停止っぽい作品も混在する。**そのまま扱い、各話/メタ/公式リンクが取れない場合は欠損を正直に出す**（「取得できませんでした」・取れたメタと生存リンクだけ表示）。誤魔化さない・壊さない。
- **変更検知（壊れたら気づける）**: 上流（snapshot・チャンネル RSS・nvapi・list.json/programlist.json・period HTML）の仕様や構造が変わったら
  **取得時に検知して失敗させる**。必須フィールド欠落・件数の急減/0・構造変化・空レスポンスを**サイレントに公開しない**。非公式源は特に厳守。方針は L2 §3/§5。
