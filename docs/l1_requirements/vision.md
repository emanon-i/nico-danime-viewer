# dアニメストア ニコニコ支店ビューア 要件・意思決定ドキュメント

> L1（要件・意思決定）。L0（`docs/l0_ideas/requirement.md`）の要求を構造化したもの。
> 技術詳細は L2、機能の受け入れ条件は L3 で確定する。

## 1. プロダクトの意思と背景

### 1.1 解決したい課題・ペイン

dアニメストア ニコニコ支店（docomo 本店とは別サービス＝ラインナップもプレイヤーも別）の公式 UI は見づらく、
「観たい・観るべき作品に出会う」までの導線が弱い。視聴可能な作品があっても発見されないまま埋もれる。
ニコニコの公開 API でメタデータ（再生数・タグ・ジャンル・投稿時間）は取得できるため、
見やすい発見サイトを軽量な静的サイトとして作る。

### 1.2 想定ユーザー／ペルソナ

- ニコニコ支店の加入者（または検討者）で、「何を観るか」をその場で発見したいライト〜ミドル層。
- ランキング・タグ・ジャンル・クールからザッピング的に作品を探したいユーザー。
- スマホ／PC ブラウザの両方で軽快に閲覧したいユーザー。

### 1.3 ビジネス上のゴール

- 個人による非営利・非公式プロジェクト。収益化しない。
- 支店ラインナップの発見性を高め、視聴は公式プレイヤーへ送客（deep-link）して公式体験を補完する。
- 公開 API のマナー（User-Agent 必須・低頻度・非配布）を守り、持続可能に運用する。

### 1.4 このプロダクトで大事にすること

- **予測どおりに動く**：同じ操作で同じ結果。ランキングや並び替えに隠れた重み付けをしない。
- **時間を奪わない**：自動再生・無限スクロールの罠・ダークパターンをしない。素早く目的に到達できるようにする。
- **迷わせない**：探し方（ランキング/タグ/ジャンル/クール）を分かりやすく分ける。
- **見やすさの理由を説明できる**：思いつきでなく、上記に沿っているかで判断する。

## 2. 制約条件・前提

### 2.1 動作環境

- 静的サイト（ブラウザ。スマホ／PC 両対応）。サーバサイドの常時稼働を持たない（GitHub Pages 配信）。

### 2.2 法規制・コンプライアンス

- 公開・非営利利用が前提。snapshot 検索 API は **User-Agent 必須・低頻度**。
- 作品データのキャッシュ／取得結果を**再配布しない**（生成 JSON は git 管理外）。
- 動画・字幕等の本体は扱わない。視聴は公式プレイヤーへの **deep-link のみ**（支店本編は有料チャンネル動画）。

### 2.3 外部システム連携

- カタログ主軸: `site.nicovideo.jp/danime/static/data/list.json`（全作品）、`programlist.json`（今季）、
  `anime.nicovideo.jp/period/<年>-<季>-danime.html`（クール・過去季）。
- メタデータ: ニコニコ snapshot 検索 API
  `https://snapshot.search.nicovideo.jp/api/v2/snapshot/video/contents/search`
  - `q=dアニメストア` / `targets=tagsExact` → レスポンスの `channelId == 2632720`（支店）でクライアント側絞り込み
    （`channelId` は filter 不可・取得のみ可）。
- ブラウザ直 API は CORS 不可 → ローカル/CI の fetch スクリプトで静的 JSON 化してから静的サイトが読む。

## 3. 要求と意思決定

### v1（今回作る範囲）

#### 全作品ブラウズ

- **ID**: REQ-0001 / **優先度**: 高
- **背景・動機**: 支店ラインナップ全体を一望できる入口が薄く作品が埋もれている。まず「全作品を見やすく並べる」基盤があることで、ランキング・タグ・ジャンル・クールの発見軸の土台になる。網羅性自体がユーザーの安心感と回遊の起点になるため最優先。

全作品を一覧・スクロール／検索・快適にブラウズできる画面。カタログ（list.json）＋ snapshot API で構築。

#### 人気ランキング（累計＋勢いスコア）

- **ID**: REQ-0002 / **優先度**: 高
- **背景・動機**: 「人気＝とりあえずの安全な選択肢」を求める行動は強く、発見の中核。snapshot API から `viewCounter`（累計）が取れ、低コストで“人気順”を予測どおりに提供できる。迷ったユーザーに最初の一歩を与える。

2 系統を提供する:
- **累計再生数順**: `viewCounter` の降順（直接構築可）。
- **勢い順（蓄積なし近似）**: `viewCounter` ÷ 投稿からの経過日数 等の**近似スコア**で「いま勢いがある」感を出す（真の期間デルタではない＝「勢い（目安）」として扱う。具体 formula は L3）。

> 検証結論（`docs/l2_foundation/validation/api-availability.md` 2回目）: 支店スコープの正確な週/月/急上昇デルタは API から直接取れず、
> かつ**自前の `viewCounter` 時系列蓄積は採用しない**ため、勢いは蓄積なし近似で代替する。

#### ジャンル別ブラウズ

- **ID**: REQ-0003 / **優先度**: 中
- **背景・動機**: 「気分・好みのジャンルから探したい」探索行動に応える軸。ランキングとは別の入口として回遊の幅を広げる。

ジャンル単位の絞り込み・一覧。<!-- 注: snapshot の `genre` は粗いため tags/categoryTags を主軸にする（L2 §6 参照） -->

#### クール（季節）別ブラウズ

- **ID**: REQ-0004 / **優先度**: 中
- **背景・動機**: アニメ視聴者は「今季」「あの頃」という季節軸での想起・探索が強い。`startTime` および補助データ（period／programlist）でクール単位に束ね、時間軸の発見導線を提供する。話題性と懐古の両方を拾う。

放送クール単位のブラウズ（今季は programlist、過去季は period ページを補助に）。

#### タグ・ブラウズ

- **ID**: REQ-0008 / **優先度**: 中
- **背景・動機**: 作品メタには豊富なタグが付き、ジャンルより細かい嗜好の手がかりになる。タグで辿れることで、ユーザー自身が探索軸を選べる能動的な発見を支える。

タグによる絞り込み・一覧（ユーザーが選んだタグで作品を探す）。

#### 新着一覧（最新の動画が軽く並ぶ）

- **ID**: REQ-0010 / **優先度**: 中
- **背景・動機**: 「今 追加された最新話」を軽く眺めたい欲求がある。トップなどに最新の動画が並ぶ枠があると、定期的に覗く動機になる。理想は毎時更新だが、メタデータ源（snapshot）の索引は日次更新のため、より高頻度な源が無ければ日次（投稿時間ベース）で出す（制約は L2 §3 参照）。

最新の動画が軽く並ぶ枠（トップのセクション＋一覧の新着順状態）。`startTime` 降順を基本に、可能なら高頻度源で毎時〜半日更新。
なお新着は **新着シリーズ（シリーズ初出）と最新話（話単位）の 2 系統**に割れる（画面/データ詳細は L2 §2.4 / screens.md）。

#### 公式プレイヤーへの deep-link

- **ID**: REQ-0005 / **優先度**: 高
- **背景・動機**: 本ツールは発見に特化し、視聴は公式に委ねる方針。各作品から公式プレイヤーへ直接遷移できることで「見つけたらすぐ観られる」体験を完結させつつ、本体を扱わない非配布原則を守れる。発見から視聴への摩擦を最小化する。

`contentId` / series id から公式プレイヤー（`nicovideo.jp/watch/<id>`・`/series/<id>`）へ deep-link。

### 将来（v1 スコープ外）

#### 急上昇（正確な期間デルタ）

- **ID**: REQ-0006 / **優先度**: 低（スコープ外）
- **背景・動機**: 「今 伸びているもの」を**正確に**出すには `viewCounter` の時系列差分（伸び率）が要る。だが検証の結論として、支店スコープでこれを直接取れる API は無く、**自前の時系列蓄積は保守・容量コストから採用しない**。本ビューアでは**蓄積なしの近似「勢いスコア」（REQ-0002）で代替**する。

正確な期間デルタ／伸び率の急上昇、および「自分のアーリー度」系は、**視聴履歴を集める別プロジェクト（ブラウザ拡張）の領分**＝本ビューアの対象外。近似の「勢い」は REQ-0002 でカバー。

#### 近い作品の提示（リコメンド）

- **ID**: REQ-0009 / **優先度**: 低（将来）
- **背景・動機**: 「これが好きなら次はこれ」を出せれば発見価値が高い。ただし根拠を説明できる（タグ・ジャンル類似など透明な）形に限る。v1 のブラウズ軸を固めてから検討する。

タグ／ジャンル類似にもとづく、根拠の見える「近い作品」提示。**v1 では作らない。**

#### 視聴履歴を使った個人化

- **ID**: REQ-0007 / **優先度**: 低（別プロジェクト依存・スコープ外）
- **背景・動機**: 「続きから／全話視聴済み」などは便利だが、**視聴履歴を集める別プロジェクト（視聴履歴ブラウザ拡張など）に依存する**。本ビューア単独では実現できず、その別プロジェクトが出来てからの話。

視聴履歴にもとづくおすすめ・続きから等。**本ビューアの対象外（別プロジェクトの産物に依存）。**

## 4. 非機能要求（高レベル）

- **挙動の一貫性**: ランキング・並び順・フィルタは「同じ操作で同じ結果」。隠れた重み付け・自動再生をしない。
- **API マナー**: User-Agent 必須・低頻度。取得は fetch スクリプト側に集約。
- **配信形態**: 静的サイト（CDN/静的ホスティングで配信可能なこと）。
- **パフォーマンス**: スマホでも軽快に一覧表示。<!-- TODO: 作品数規模・初期表示目標はL2で確定 -->
- **更新頻度**: 発見系（カタログ/ランキング）は**日次で十分**。**新着は可能なら毎時〜半日**で更新したい。
  ただし snapshot の索引は日次更新のため、毎時の新着には別の高頻度源が要る（無ければ日次）。方針は L2 §3。
- **変更検知（壊れたら気づける）**: 上流（API・list.json/programlist.json・period HTML）の仕様やページ構造が変わったら
  **取得時に検知して失敗させる**。必須フィールド欠落・件数の急減/0・構造変化・空レスポンスを**サイレントに公開しない**。方針は L2 §3/§5。
