## 1. プロダクトの意思と背景

### 1.1 解決したい課題・ペイン

dアニメストア ニコニコ支店（docomo 本店とは別サービス＝ラインナップもプレイヤーも別）の公式 UI は見づらく、
「観たい・観るべき作品に出会う」までの導線が弱い。視聴可能な作品があっても発見されないまま埋もれる。
ニコニコの公開 API でメタデータ（再生数・タグ・ジャンル・投稿時間）は取得できるため、
自分にとって見やすい発見サイトを軽量な静的サイトとして自作したい。

### 1.2 想定ユーザー／ペルソナ

- ニコニコ支店の加入者（または検討者）で、「何を観るか」をその場で発見したいライト〜ミドル層。
- ランキング・タグ・ジャンル・クールからザッピング的に作品を探したいユーザー。
- スマホ／PC ブラウザの両方で軽快に閲覧したいユーザー。

### 1.3 ビジネス上のゴール

- 個人による非営利・非公式プロジェクト。収益化しない。“プラットフォーム”ではなく“自分のための道具”。
- 支店ラインナップの発見性を高め、視聴は公式プレイヤーへ送客（deep-link）して公式体験を補完する。
- 公開 API のマナー（User-Agent 必須・低頻度・非配布）を守り、持続可能に運用する。

### 1.4 このプロダクトで大事にすること（北極星）

> **「最小驚きの原則 ＋ ユーザー主権」**

- ランキング・検索・並び替えが**予測どおりに動く**（Principle of Least Astonishment／Nielsen ヒューリスティクス／Hick の法則）。
- **アテンション経済の逆を行く**：滞在時間最適化・不透明アルゴリズム・ダークパターン・自動再生をしない。
  ユーザーの時間を奪わず、**素早く目的に到達させて離脱させる**ことを良しとする。
- 「見やすさ」は主観でなく**行動心理の原理で根拠づける**。データと挙動はユーザーの側に置く（ユーザー主権）。

## 2. 制約条件・前提

### 2.1 動作環境

- 静的サイト（ブラウザ。スマホ／PC 両対応）。サーバサイドの常時稼働を持たない（GitHub Pages 的配信を想定）。

### 2.2 法規制・コンプライアンス

- 公開・非営利利用が前提。snapshot 検索 API は **User-Agent 必須・低頻度**。
- 作品データのキャッシュ／スクレイピング結果を**再配布しない**（生成 JSON は git 管理外）。
- 動画・字幕等の本体は扱わない。視聴は公式プレイヤーへの **deep-link のみ**（支店本編は有料チャンネル動画）。

### 2.3 外部システム連携

- 主軸: ニコニコ snapshot 検索 API
  `https://snapshot.search.nicovideo.jp/api/v2/snapshot/video/contents/search`
  - `q=dアニメストア` / `targets=tagsExact` → レスポンスの `channelId == 2632720`（支店）でクライアント側絞り込み
    （`channelId` は filter 不可・取得のみ可）。
- 補助: `site.nicovideo.jp/danime/static/data/list.json`（全作品）、`programlist.json`（今季）、
  `anime.nicovideo.jp/period/<年>-<季>-danime.html`（クール・過去季）。
- ブラウザ直 API は CORS 不可 → ローカル/CI の fetch スクリプトで静的 JSON 化してから静的サイトが読む。

## 3. 要求と意思決定

### 全作品ブラウズ

- **ID**: REQ-0001 / **優先度**: 高
- **背景・動機**: 支店ラインナップ全体を一望できる入口が薄く作品が埋もれている。まず「全作品を見やすく並べる」基盤があることで、ランキング・タグ・ジャンル・クールの発見軸の土台になる。網羅性自体がユーザーの安心感と回遊の起点になるため最優先。

全作品を一覧・スクロール／検索・快適にブラウズできる画面。snapshot API ＋ 補助カタログ（list.json）で構築。

### 再生数ランキング

- **ID**: REQ-0002 / **優先度**: 高
- **背景・動機**: 「人気＝とりあえずの安全な選択肢」を求める行動は強く、発見 UI の中核。snapshot API から `viewCounter` が取れ、低コストで予測どおりの“人気順”を提供できる。迷ったユーザーに最初の一歩を与える。

`viewCounter` の降順ランキング表示。

### ジャンル別ブラウズ・近いジャンル提示

- **ID**: REQ-0003 / **優先度**: 中
- **背景・動機**: 「気分・好みのジャンルから探したい」探索行動に応える軸。さらに「近いジャンル」を提示することで、最小驚きを保ちつつ回遊の幅を広げる。嗜好ベースの発見を可能にする。

ジャンル単位の絞り込み・一覧と、近接ジャンルへの導線。<!-- TODO: 「ジャンル」の定義はL2で確定（後述のデータ可用性参照） -->

### クール（季節）別ブラウズ

- **ID**: REQ-0004 / **優先度**: 中
- **背景・動機**: アニメ視聴者は「今季」「あの頃」という季節軸での想起・探索が強い。`startTime` および補助データ（period／programlist）でクール単位に束ね、時間軸の発見導線を提供する。話題性と懐古の両方を拾う。

放送クール単位のブラウズ（今季は programlist、過去季は period ページを補助に）。

### 公式プレイヤーへの deep-link

- **ID**: REQ-0005 / **優先度**: 高
- **背景・動機**: 本ツールは発見に特化し、視聴は公式に委ねる方針。各作品から公式プレイヤーへ直接遷移できることで「見つけたらすぐ観られる」体験を完結させつつ、本体を扱わない非配布原則を守れる。発見から視聴への摩擦を最小化する。

`contentId` / series id から公式プレイヤー（`nicovideo.jp/watch/<id>`・`/series/<id>`）へ deep-link。

### タグ・ブラウズ＋お気に入りタグからの開拓

- **ID**: REQ-0008 / **優先度**: 中
- **背景・動機**: 作品メタには豊富なタグが付き、ジャンルより細かい嗜好の手がかりになる。タグで辿れること、さらに「お気に入りタグ」から未見の新しい作品を開拓できることで、ユーザー主権の能動的な発見を支える。レコメンドの不透明さを避け、ユーザー自身が探索軸を選べる。

タグによる絞り込み・一覧と、指定タグ群からの未見作品サジェスト。

### 賢いリコメンド（近い作品提示）

- **ID**: REQ-0009 / **優先度**: 中
- **背景・動機**: 「これが好きなら次はこれ」を予測どおりに提示できれば発見価値が高い。ただしアテンション最適化の不透明アルゴリズムは北極星に反するため、**根拠の説明できる（タグ・ジャンル類似など透明な）**リコメンドに限る。ユーザーが納得できる近接作品の導線を作る。

タグ／ジャンル類似にもとづく、根拠の見える「近い作品」提示。<!-- TODO: 類似度ロジックはL2/L3で具体化 -->

### 急上昇（人気の伸び）

- **ID**: REQ-0006 / **優先度**: 低（phase2）
- **背景・動機**: 累計再生数では「今 伸びているもの」を捉えられず旬の発見が弱い。`viewCounter` を日次スナップショットして差分（伸び率）で算出できるが、スナップショット蓄積基盤の運用が必要なため phase2。

`viewCounter` の日次差分による急上昇ランキング。

### 個人化（ローカルファースト視聴履歴連携）

- **ID**: REQ-0007 / **優先度**: 低（phase2）
- **背景・動機**: 「続きから／全話視聴済み」などのパーソナライズは回遊価値を高めるが、履歴の取得・保持はユーザー主権／プライバシー設計が要。ActivityWatch 型のローカルファースト連携を想定し、v1 の発見軸を固めてから着手する。

視聴・閲覧履歴（ローカル保持）にもとづくおすすめ・続きから等。<!-- TODO: 履歴の取得経路・保持方式は要確認 -->

## 4. 非機能要求（高レベル）

- **北極星の遵守**: 各機能は「予測どおり／時間を奪わない／根拠が説明できる」を満たすこと（§1.4）。
- **API マナー**: User-Agent 必須・低頻度。取得は fetch スクリプト側に集約。
- **配信形態**: 静的サイト（CDN/静的ホスティングで配信可能なこと）。
- **パフォーマンス**: スマホでも軽快に一覧表示。<!-- TODO: 作品数規模・初期表示目標はL2で確定 -->
- **データ鮮度**: fetch をローカル/CI で定期実行し `data/*.json` を更新（索引は毎日 AM5:00 更新）。<!-- TODO: 更新頻度は要確認 -->
